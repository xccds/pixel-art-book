# 练手04 明度：去色之后才见真章

### 4.0 这一章解决什么问题

练手02 你给贯穿角色长出了体积——纸片变成了有厚度的东西，单光源下有了亮面、中间调、暗面。练手03 你给场景安排了空间——正负形、纵深五层、呼吸点。但只要你把角色放进场景，画面还是"不对劲"：角色溶进背景、焦点不知道在哪、整张图灰蒙蒙的"糊"。问题不在颜色，不在形状，**在明度（Value）——画面里最被你忽略、却最该先做对的那一层。**

观察02 把"明度"列为八种数据类型之一，给了你一句话定义和两个极端：《Downwell》黑底白敌红道具，几乎只靠明度对比；《星露谷》靠季节色相编码，去色后部分信息丢失。这一章把它彻底拆开：明度为什么是骨架、为什么程序员天然擅长、怎么用三区四模板给画面定基调、怎么用九级明度阶和眯眼测试自检、怎么走"灰度先于色彩"的工作流、以及怎么用明度引导玩家视线和情绪。

这一章是**第二部最重要的章节**——书 A 把它定为明度专章，书 B 把它作为"颜色的骨架"间奏章专门讲。两本合并后，A 的骨架/去色/三区四模板/灰度缩略图/三点光照打底，B 的九级明度阶/灰度→色彩工作流/明度引导与情绪接上"可读性与情绪"那一半。一块拼成一条流水线：**去色诊断（看得出问题）→ 选模板定基调（决定情绪）→ 灰度先于色彩（先建骨架）→ 三点光照执行（打光）→ 引导视线（控焦点）**。

你只需要 Aseprite，加一支笔一张纸。

### 4.1 程序员视角：明度是骨架

#### 明度 = 你每天都在处理的那个整数

打开你的代码编辑器，想象你定义了一个颜色变量：

```
let playerHue = "red";        // 色相：什么颜色
let playerSaturation = 0.8;   // 饱和度：颜色多纯
let playerValue = 0.6;        // 明度：有多亮
```

如果你熟悉 HSV/HSL 色彩模型，明度就是这个模型的第三个分量。但大多数人只盯前两个——"我要一个深红色"——然后随手给一个 V 值。这就是问题的开始。

**明度（Value）**：一个颜色去色之后呈现的灰度深浅。纯白 = V=100%，纯黑 = V=0%。它是色彩三要素里唯一一个独立于"颜色感"的属性——你可以把明度理解为每个像素的"亮度信息"。灰度值（Gray Value）就是一个 0–255 的整数，画面的明度结构本质上是**一个二维矩阵的数值分布问题**。

> **程序员类比 #1：** 明度之于画面 = 变量的实际值（Value）之于类型标签（Type）。你可以在代码里随意改一个变量的类型标签——`int` 改 `float`、`"red"` 改 `"blue"`——但如果变量的实际数值算错了，换什么类型都救不回来。同理，明度决定了画面的**结构性正确**——色彩只是"类型标签"，明度才是"值"。值错了，结构崩了，标签再好看也没用。

#### 为什么程序员天然擅长明度

因为你每天都在处理"值"。你可以用你熟悉的方式去理解明度：

- **明度分布 = 直方图（Histogram）**。亮区多 → 画面亮，暗区多 → 画面暗。
- **明度对比 = 梯度（Gradient）**。相邻区域明度差越大 → 边界越锐利，视觉冲击越强。
- **明度分层 = 图层（Layer）**。前景、中景、远景各占一个明度区间 → 空间感自动产生。

人眼对明度差异的敏感度远高于对色相差异的敏感度——这是进化结果，祖先需要在草丛里分辨"略暗的一块"（可能是捕食者），而不是判断"那是哪种绿色"。在游戏画面里这意味着：**玩家在 0.3 秒内通过明度差异定位交互目标，而不是通过颜色。** 你的红色攻击按钮之所以显眼，首先是因为它比周围亮（或暗），其次才是因为它是红色的。你不需要"艺术天赋"来掌握明度，你需要的是一个系统化的思维框架。

### 4.2 去色测试——程序员的"美术 debug 模式"

找一个你觉得配色绝佳的游戏截图，用任何图像软件点一下"去色"（Photoshop / Krita 里是 Ctrl+Shift+U，Aseprite 里见 4.5 的复制窗口法）。变成黑白后，如果画面还是一团模糊——恭喜，你发现了问题的根源：你一直在用色彩掩盖明度的混乱。

James Gurney 在《Color and Light》里写下一句被无数美术师引用的论断 [1]：

> "If your picture looks bad in black and white, it won't be saved by color."
> （如果你的画面在黑白下不好看，加颜色也救不回来。）

Gurney 指出，专业美术师在创作彩色作品前，几乎都会先用单色（grisaille）建立完整的明度结构，色彩是最后一步。这个工作流在传统油画里存在了几个世纪，在现代数字绘画里依然有效。

![去色前后对比](images/练手/练手04-fig03-去色前后对比.svg)

*图 练手04.3：同一张画面彩色（上）与去色（下）。去色后焦点还在、层次还清 = 明度扎实；去色后糊成一团 = 你在用颜色作弊。*

> **程序员类比 #2：** 去色检验 = 排除变量 debug。你遇到一个 bug，页面渲染错误，你知道可能是 CSS 的锅也可能是 JS 数据的锅——你做的事情是**禁用 CSS，先看裸 HTML 结构是否正确**。去色就是"禁用色彩"，只测明度骨架。如果黑白下结构清晰，加颜色后只会更好；如果黑白下已经糊了，那就别碰颜色——先修明度。

读完这一章，你将获得一个终身受用的技能：在任何图像编辑器里按下去色按钮，3 秒内判断一张图的明度结构是否正确。这就是程序员的"美术 debug 模式"。

### 4.3 三区四模板——给画面定基调

#### 明度三区系统（Value Three-Zone System）

把画面简化成三个明度区域，这是最基础的诊断工具：

- **亮区（High Key）**：明度范围 7–10（以 0–10 为全范围）。大面积亮区 → 明亮、轻快、开阔、安全。
- **中调区（Mid Tone）**：明度范围 3–7。画面信息最密集的区域，主角、主要交互元素通常位于此。
- **暗区（Low Key）**：明度范围 0–3。大面积暗区 → 神秘、压抑、恐怖、危险；小面积暗区提供深度和重量。

![明度三区系统](images/练手/练手04-fig01-明度三区.svg)

*图 练手04.1：亮区/中调区/暗区。关键不是每区什么颜色，而是每区的**面积占比**——同一构图，改三区占比就改了情绪。*

#### 四种主流明度模板

经过对上百款独立游戏画面的分析，画面的明度分布可归纳为四种模板：

- **高调（High Key）**：约 80% 区域为亮区。适合明亮户外、UI、休闲游戏。范例：《星露谷物语》白天、《旅途》沙漠。情绪：轻松、开放、安全。
- **低调（Low Key）**：约 60% 以上区域为暗区。适合恐怖、潜行、地下城。范例：《Limbo》《Inside》[6]。情绪：压抑、神秘、恐惧。
- **高对比（High Contrast）**：亮区和暗区都占显著面积，亮暗极端并置。适合 Boss 战、高潮、剧情节点。范例：《茶杯头》Boss 战、《哈迪斯》战斗高潮。情绪：戏剧性、紧张、冲击。
- **低对比（Low Contrast）**：大部分区域集中在中调区，没有极端亮暗。适合梦境、回忆、抒情。范例：《Gris》回忆、《奥日》抒情段落。情绪：柔和、朦胧、内省。

![四种明度模板](images/练手/练手04-fig02-四模板.svg)

*图 练手04.2：四种明度模板的灰度示意与代表游戏。选定一个模板作为画面基调，再在局部做变化。*

这些模板不是死规则，而是"起点模板"。你选定一个作为画面明度基调，然后在局部做变化。Nathan Fowkes（梦工厂美术指导）的方法是 [2]：给前景、中景、远景、背景各分配一个明度区间，画出**四个明度图层**——层与层之间保持明度差异，层内可变化但不可越界。

#### 面积比例量化：16 格估算法

"大面积亮区"这种描述太模糊。下面是每个模板的**参考面积比例**——不是精确公式，是帮你做第一轮自检的估算起点。

**如何快速估算：** 1) 把画面去色（Ctrl+Shift+U）；2) 眯起眼睛，画面会自动合并成几个大的灰度块；3) 把画面在心里分成 4×4=16 个格子，对每格判断"偏亮/偏暗/中调"；4) 数亮格/暗格/中调格的数量 ÷ 16 → 近似面积百分比。

| 模板 | 亮区占比 | 中调区占比 | 暗区占比 | 情绪 | 适合场景 |
|------|---------|----------|---------|------|---------|
| 高调 | 60–80% | 20–35% | 0–5% | 明亮、轻快、安全 | 白天户外、休闲 UI、新手关卡 |
| 低调 | 0–5% | 25–35% | 60–75% | 压抑、恐怖、神秘 | 地下城、恐怖关卡、潜行关卡 |
| 高对比 | 30–40% | 20–30% | 30–40% | 戏剧性、紧张、冲击 | Boss 战、剧情高潮、PVP 界面 |
| 低对比 | 5–15% | 75–90% | 5–15% | 柔和、梦幻、内省 | 回忆、抒情、过场动画 |

**用法：** 先确定你要的情绪 → 选模板 → 按比例画灰度缩略图（4.6）→ 用 16 格粗估法检查实际比例是否在参考范围内。偏离太远不是"你的风格独特"，是"你的画面可能没传达你想传达的情绪"。这不是设计公式，是训练用的自检工具——等你对明度的直觉足够强，你的眼睛会在 3 秒内告诉你"这个画面的明度结构对不对"。但在这之前，数格子给了你一条"对错"参考线——对自学者来说这是无价的。

### 4.4 九级明度阶 + 眯眼测试

书 B 把明度拆得更细：传统美术用 **9 级明度标尺** [3]，从 0（纯白）到 9（纯黑）。像素画里通常只用其中 4–6 级。

![九级明度阶](images/练手/练手04-fig06-九级明度阶.svg)

*图 练手04.6：九级明度阶（0=纯白 → 9=纯黑），附眯眼测试提示。像素画常用 4–6 级，每级对应一个形体功能。*

| 级别 | 名称 | 用途 |
|------|------|------|
| 0–1 | 高光（Highlight） | 反光最强处，面积最小 |
| 2–3 | 亮面（Light） | 正面光源照射的区域 |
| 4–5 | 中间调（Midtone） | 物体的"真实颜色"区域 |
| 6–7 | 暗面（Shadow） | 背光区域 |
| 8–9 | 闭塞阴影（Core Shadow） | 光线完全无法到达的区域 |

**像素画的明度经验法则：** 每个颜色阶梯（Color Ramp）跨越 2–4 级明度即可；高光和阴影之间明度差至少 3 级，否则体积感出不来；不同物体之间明度差至少 1 级，否则它们会连在一起。一个好的 5 级色板阶梯在明度上大致是：`亮面(2) → 中间偏亮(3.5) → 中间调(5) → 中间偏暗(6.5) → 暗面(8)`——**分布要均匀**，阶梯里有两步明度几乎一样，那两步就是浪费的，可以合并。

> **程序员类比 #3：** 九级明度阶 = 把连续的 0–255 灰度值域**量化（quantize）**成 9 个离散档位，就像音频的位深采样——你不需要连续值，9 个档位足够重建任何形体。像素画只用 4–6 级 = 你只取枚举里最关键的几个值。明度分布均匀 = 数组步长均匀；两步明度几乎一样 = 冗余索引，合并掉。

#### 眯眼测试（Squint Test）

这是每个像素艺术家都该养成的习惯，和 4.2 的去色测试是一对互补的自检法：

1. **眯起眼睛**看你的精灵（或退远两步看）；
2. 如果精灵的各个部分无法清晰区分——它们明度太接近；
3. 如果角色和背景融为一体——它们明度缺乏对比。

一个亮黄的角色和一个深紫的背景在明度上有巨大差异，即使完全褪色成灰度也能清晰分离；而两个不同色相但明度接近的颜色（如中蓝和中绿），在灰度下几乎无法区分。眯眼和去色做的是同一件事——**把色相这个"高频噪音"滤掉，只看明度这个"低频结构"**。

> **程序员类比 #4：** 眯眼测试 = 给图像跑一个**低通滤波 / 降采样（downsample）**。眯眼模糊掉高频细节（色相、纹理、锐边），只留低频的明度结构——就像给图像套一层 Gaussian blur 看结构是否还在。模糊后还立得住的画面 = 明度结构扎实；模糊后塌成一坨 = 你在靠高频细节撑场面。

### 4.5 灰度→色彩工作流——先写 HTML，再加 CSS

"先定明度，再上颜色"是专业像素艺术家最核心的工作流之一。三步：

**第一步：用灰度画"素描"。** 在 Aseprite 里新建文件，只用灰度颜色（R=G=B）画草图，不管颜色只管明暗。用 3–4 级灰度就够：深灰（轮廓/阴影）、中灰（主体）、浅灰（亮面）、白（高光）。这一步的目标：**在灰度下你的精灵已经清晰可辨、有体积感。**

**第二步：叠加颜色。** 在灰度草图之上新建图层，混合模式设为 **Overlay（叠加）** 或 **Color（颜色）**，然后用你选的配色方案涂色。颜色层只改色相和饱和度，不动底层的明度结构。

**第三步：精修。** 灰度层和颜色层合并后，手动微调不满意区域的色彩和明暗。

> **程序员类比 #5（本节核心）：** 灰度→上色 = **先写 HTML 结构，再加 CSS 样式**。灰度草图是 HTML——它定义画面的**结构**（明度关系、层次、焦点）；Overlay/Color 图层是 CSS——它定义画面的**样式**（色相、饱和度）。结构（明度）对了，样式（色彩）怎么改都不会塌——你换一套配色就像换一套主题 CSS，骨架不动。反过来如果先画彩色再调明度，等于在没写 HTML 的情况下硬塞 CSS，改一个样式整个布局就崩。**这个工作流强制你在最影响可读性的维度（明度）上先做对，再考虑色彩表现。**

![灰度到色彩工作流](images/练手/练手04-fig07-灰度到色彩工作流.svg)

*图 练手04.7：灰度草图（HTML 结构）→ Overlay/Color 叠加（CSS 样式）→ 精修。结构对了，换多少套配色都不会塌。*

#### Aseprite 里实时看灰度

不必每次去色再切回来——Aseprite 有更顺手的法子：**Window → Duplicate View（复制窗口）**，新窗口设为灰度预览，两个窗口实时同步——一个看彩色、一个看灰度，边画边验。或者更简单：眯起眼睛看（见 4.4）。即使你不走完整的灰度→上色工作流，也应该养成一个习惯——**每画一段时间就切到灰度模式检查明度关系**。

### 4.6 灰度缩略图——30 秒 / 2 分钟 / 5 分钟

在真正动手画之前，先用缩略图（Thumbnail）测试明度结构。缩略图的目标不是"画得像"，是用最小投入验证画面骨架是否成立。三个版本，按手头时间选：

**30 秒速写版（只用 3 个灰度色块）：** 在 200×150 小画布上，用 #FFFFFF（白）、#808080（中灰）、#000000（黑）画明度分区——白=最亮、黑=最暗、灰=中间调。眯眼看：亮/灰/暗各占多少？焦点区是否是亮暗对比最强处？30 秒后你还不知道焦点在哪——这张图的明度结构不成立，回去重分三区。

**2 分钟标准版（3–5 个灰度层级）：** 在 400×300 画布上，从 #FFF/#CCC/#999/#666/#333/#000 里选 5 个，快速铺前/中/远景明度分区——前景最暗或最亮（做框架）、中景对比最强（焦点）、远景对比最低（推远）。合格标准：去色后前中后三层能眯眼分辨。

**5 分钟精细版（完整灰度渲染）：** 在标准画布上用完整 0–255 灰度，画三步照明（4.7）+ 大气透视渐变 + 暗角。这张灰度图就是你接下来的"骨架层"——在它上面叠颜色时用"正片叠底"或"叠加"混合模式，颜色永远不会破坏你的明度结构。合格标准：把这张图给一个没看过你设计的人，让 Ta 说出焦点在哪——3 秒内指对，明度结构成立。

三种版本背后是同一个原则：**先建骨架，再穿衣服。** 骨架可以在 30 秒内验证，也可以在 5 分钟内精细搭建。但无论投入多少时间，步骤不能反过来——不要在已经有颜色的画面上"修明度"，那就像在已经编译好的二进制上改源代码。

### 4.7 三点光照（Three-Point Lighting）

来自电影和摄影的标准技法 [4]，适用于任何引擎和任何画风：

- **Key Light（主光源）**：画面中最强的光源，决定主要明度结构。通常放在主体右上方或左上方 45°。
- **Fill Light（补光）**：来自主光源对面的柔光，减弱阴影但不是消除阴影。强度通常只有 Key 的 30%–50%，可以用冷色（向蓝色偏移）增加色彩趣味。
- **Rim Light（轮廓光）**：来自主体后方的光线，在主体边缘产生一条亮边，将主体从背景中分离出来——这是"让角色跳出画面"的最有效技法。

![三点光照](images/练手/练手04-fig04-三点光照.svg)

*图 练手04.4：Key 决定明度结构（主）· Fill 控制暗部细节（辅）· Rim 分离轮廓（点睛）。*

公式：**Key 决定明度结构 · Fill 控制暗部细节 · Rim 分离轮廓**。你可以在任何引擎实现这三步——对应三个光源或三个光照层。2D 游戏里就是三个叠加的渐变图层，像素艺术里就是三个调色板分区。你在练手02 L1 已经给角色打过单光源（Key），这一章加上 Fill 和 Rim，角色的明度结构才完整。

### 4.8 明度引导视线与情绪

#### 用明度创造焦点

明度是控制玩家视线最有力的工具——人眼本能地被高对比度区域吸引 [5]。在一个场景中，**明度最高且与周围对比最大的区域**就是视觉焦点。这就是"高光法则"：你的画面里，最高明的区域只留给一个地方——主角的脸或关键交互点。背景元素降低明度对比，避免抢戏。明度还能创造深度——大气透视从明度角度理解就是：远处物体明度更高、对比度更低、细节更少（前景低明度高对比、中景中间、背景高明度低对比，见练手03 的纵深五层）。

> **程序员类比 #6：** 明度焦点 = DOM 里 **z-index 最高 / opacity 最满**的节点。玩家目光默认落到权重最高的明度区块，就像浏览器先把最高 z-index 的元素渲染到最上层。"高光法则" = 一个画面只允许一个最高 z-index 元素，多了就打架，玩家的视线不知道该往哪落。

#### 明度与情绪：四种明度调式

同一个角色在不同明度模式下传达完全不同的情绪——这正是 4.3 四模板的情绪侧展开。传统美术把这叫"明度调式"：

| 明度模式 | 效果 | 适用场景 |
|---------|------|---------|
| **高长调**（大面积亮 + 小面积暗，反差大） | 明快、清晰、活力 | 白天关卡、UI |
| **低长调**（大面积暗 + 小面积亮，反差大） | 戏剧性、紧张、神秘 | 地牢、夜晚、恐怖 |
| **中间调**（明度范围窄，反差小） | 柔和、梦幻、怀旧 | 黄昏、雾天、回忆场景 |
| **高短调**（整体亮，反差小） | 轻盈、柔和、空灵 | 天堂、云端、梦境 |

> **程序员类比 #7：** 这四种明度调式 = 四套**明度配置 preset**（像应用的 dark / light / 高对比 / 柔和模式）。你切一个 preset 就切了画面的情绪基调——高长调 = light + 高对比 mode，低长调 = dark + 高对比 mode。先选 preset（定情绪），再在 preset 的明度区间里画，比"凭感觉调亮调暗"可操作得多。

### 4.9 游戏中的明度分层

游戏画面和静态插画有一个本质区别：玩家需要从画面中提取**交互信息**。明度结构服务于可读性（Readability）——这和练手02 讲的角色可读性是同一件事，只是从单个精灵放大到整张画面：

| 优先级 | 元素 | 明度策略 |
|--------|------|----------|
| 1 | 玩家角色 | 最高对比度，与背景保持 ≥30% 明度差 |
| 2 | 敌人/危险 | 次高对比度，用明度突变引起注意 |
| 3 | 可交互物品 | 明度"弹出"，与周围保持明显差距 |
| 4 | 导航路径 | 用明度渐变引导视线方向 |
| 5 | 装饰/背景 | 低对比度，灰度接近，不抢夺注意力 |

![游戏明度分层](images/练手/练手04-fig05-游戏明度分层.svg)

*图 练手04.5：游戏画面的明度优先级。玩家角色 > 危险 > 可交互 > 导航 > 装饰——明度对比度按这个顺序递减。*

一个实用技巧——**暗角（Vignette）**：把画面四角压暗，自然地把玩家视线推向画面中心。几乎所有 3A 游戏都在用，实现只需要一个径向渐变叠加。

> **程序员类比 #8：** 明度是画面的架构，色彩是 UI 皮肤。架构错了（明度混乱），换什么皮肤都白搭——就像代码的底层数据流错了，前端框架换再多也没用。先修架构，再谈美化。优先级表 = 渲染队列的排序规则：高优先级元素先"抢占"明度对比度，低优先级元素主动让出，整张画面的注意力分配才不会乱。

### 4.10 练习

本章练习逐级递进，每级有明确"及格标准"——你不需要完美，达到标准就进下一级。三个练习建议分三天。

> **贯穿式角色项目（续）：** 练手01 你画了 `character-v01-outline.aseprite` 的剪影/线稿；练手02 L1 你给它长出体积、存为 `character-v02-volume.aseprite`（单光源、3 级明度、AO + 反射光）；练手03 你练的是场景，角色项目暂停一程。**本章 L1 接回角色项目**——用明度思维重新给那个角色打光：先选一个明度模板（4.3）、用九级明度阶（4.4）量化你的色板阶梯、走灰度→上色工作流（4.5）、补上 Fill 和 Rim（4.7）。这个角色还将在**练手05（色彩）**正式上色、**制作02（像素角色工作流）**收口成可导入引擎的完整角色。所以 L1 你画的不只是练习，是给你贯穿角色的"明度骨架层"。

#### L1 · 给贯穿角色重新打光（约 45-90 分钟）

**目标**：把练手02 的 `character-v02-volume.aseprite` 用明度思维重做一遍——从"凭感觉打光"升级到"按模板和九级阶打光"。

**你需要**：练手02 的 `character-v02-volume.aseprite`、Aseprite、纸笔。

**步骤**：
1. 先决定角色的明度模板：这个角色出现在什么场景？白天户外 → 高调、地牢 → 低调、Boss 战 → 高对比。在纸上写下你选的模板和理由。
2. 用九级明度阶（4.4）给角色的色板阶梯定档：高光=1、亮面=3、中间调=5、暗面=7、闭塞=9（按你的模板调整区间）。在纸上画一个小色阶条标注每级用途。
3. **先画灰度版**：在 Aseprite 复制一份 `character-v02-volume.aseprite`，去色，按九级阶重新分配明度——这是你的"HTML 结构层"。
4. 灰度版眯眼测试 + 去色检验：眯眼后角色和背景（假设的）能分离吗？焦点（脸/武器）是明度最高区吗？不合格回去调三区比例。
5. 灰度过关后，新建 Overlay/Color 图层上色——这一步只改色相饱和度，不动明度结构（"加 CSS"）。
6. 补上 Fill（暗部柔光）和 Rim（边缘亮边），完成三点光照。
7. 最后再按一次 Ctrl+Shift+U 去色——新加的颜色有没有破坏原本的明度关系？保存为 `character-v03-value.aseprite`。

**合格标准**：
- [ ] 纸上写了所选明度模板 + 理由，及九级明度阶的定档。
- [ ] 灰度版先于彩色版完成，且灰度版通过眯眼测试。
- [ ] 色板阶梯覆盖至少 4 级明度，相邻两级明度差不小于 1 级（无冗余档）。
- [ ] 三点光照齐全（Key + Fill + Rim），Rim 在角色边缘可见。
- [ ] 最终去色后焦点仍在、层次仍清——明度结构没被颜色破坏。
- [ ] 文件存为 `character-v03-value.aseprite`，准备交给练手05 上色。

#### L2 · 去色观察（约 15 分钟）

**目标**：建立"用黑白眼睛看画面"的本能。

**步骤**：找 5 张你喜欢的游戏彩色截图，用任何图像软件去色（Photoshop/Krita: Ctrl+Shift+U；GIMP: Colors→Desaturate；手机相册的黑白滤镜也行）。对每张图回答三问：①画面中最亮/最暗的部分在哪？②主角/焦点和背景之间有明显亮度差吗？③眯眼看，画面里有几个大的明度块面？

**合格标准**：能对至少 3 张图给出明确回答，而不是"好像差不多"。**预期发现**：优秀的游戏画面去色后依然层次分明——主角亮背景暗，或背景亮主角暗；差的反例则是一团灰色糊在一起。你会开始注意到，那些你觉得"配色好看"的游戏，实际上首先是"明度结构好"。

#### L3 · 灰度球体三光照（约 30-60 分钟）

**目标**：理解同一形体在不同光照方向下，明度分布如何改变形体感知。

**工具**：任何能画灰度的工具——纸+铅笔、Krita、Aseprite、甚至 iPad 备忘录。

**步骤**：画一个圆（只是圆的轮廓），用灰度把它渲染成三个球体，每个对应不同光照方向：**顶光**（亮面在顶、暗面在底、投影向下，感觉物体稳稳在地）；**侧光**（亮面在侧、暗面在对侧，感觉戏剧性有体积感）；**逆光**（主体大部分在阴影中、边缘一圈亮边 Rim Light，感觉神秘剪影感）。三个球体形状都是圆，区别只在**明度分布**——这就是"明度定义形体"的本质。

**合格标准**：三个球体能让人一眼看出光照来自不同方向，不需要标注。**进阶**：想让球体看起来是白色，亮面用 90% 灰、暗面用 30% 灰；想让球体看起来像金属，亮面用 100% 白、暗面用 5% 黑、中间加一道锐利明度分界线——材质感也是明度分布决定的（练手02 已展开，练手06 再深化）。

#### 去色检验法（核心技法）

这是本章最重要的一个技法，值得单独标出：**在任何绘画阶段，按 Ctrl+Shift+U 去色检查一次。** 理想工作流：①先画灰度缩略图定明度结构（5 分钟）；②在灰度下完成 70% 的形体塑造（主体时间）；③叠加颜色（锦上添花，不是雪中送炭）；④反复切换去色模式验证新加的颜色有没有破坏原本的明度关系。如果你发现一张图画到一半去色后"全都没了"——别灰心，这是所有自学美术的人都会经历的阶段。好消息是：你现在知道问题在哪，而修复它的方法是系统化和可操作的。

### 4.11 常见错误与诊断

**错误 1：等明度并列（最致命）。** 画面中两个重要元素（角色和背景、主角和敌人）去色后明度值几乎一样，变成一团灰色。**原因**：选色时只看色相（"这个红那个绿"），没检查灰度下的相对关系。**修复**：打开信息面板（Photoshop: F8），选中两个元素分别看 B（Brightness）值，差值小于 20 就调其中一个——角色与背景明度差建议 ≥40（经验起点，不是无障碍标准）。这和练手02 错误 3"只靠颜色区分"是同一病灶的两面：颜色是增强不是替代。

**错误 2：明度范围太窄 / 中间调泛滥（最常见的整体问题）。** 整个画面只有 2–3 级明度，发灰、没层次、没视觉冲击。**原因**：恐惧"画太亮"或"画太暗"，安全地待在中间调里——白色画布不敢画暗、黑色不敢画亮，这是初学者本能。**修复**：强制自己先铺一个极端明度——5 分钟内只用亮/中/暗三个色块铺满画面，三块里必须有一个低至 0–2 的深色和一个高至 9–10 的浅色。色板阶梯覆盖从暗到亮的完整范围，高光要亮、阴影要暗。

**错误 3：明度跳跃过大（破坏空间感的隐形杀手）。** 一个物体的阴影直接跳到最暗，没有中间过渡，看起来像生硬拼贴；或场景中远景某部分比中景还暗还抢眼。**原因**：色板阶梯两级之间跳跃超过 3 级明度没插过渡色；或画远景时用了比中景更饱和/更暗的颜色，破坏大气透视（练手03 纵深五层讲过：远景明度必须 ≥ 中景 ≥ 近景，越远越亮/灰）。**修复**：检查色板阶梯明度是否均匀分布，跳跃超 3 级就插一个过渡色；远景永远不允许出现画面中最暗或最亮的点。

**错误 4：多个高对比区域同时存在（焦点失焦）。** 画面中多个区域都有高对比度，观众视线不知道看哪——违反 4.8 的"高光法则"。**原因**：每个元素都想"跳出来"，结果互相抢戏。**修复**：全画只留一个最高对比区域（主角/焦点），其他区域降低对比度。这和练手03 的"呼吸点"是同一原则的两侧：留白让眼睛休息，单一焦点让眼睛知道往哪看。

**错误 5：零渐变突兀割裂。** 两个明度区域之间有明显硬边，像"贴上去的"。**原因**：建明度结构时用纯色块 + 硬边笔刷，没做过渡。**修复**：大明度结构定了之后，在两块交界处加柔和渐变过渡。自然界很少有绝对的明度硬边（除非投影边缘或金属高光）——但也别全用软边，那会变"棉花糖"。好画面是**大面积软过渡 + 关键位置硬边**的组合。

### 4.12 小结

明度是画面骨架。它决定了"玩家看哪里"和"画面是什么感觉"——这两件事发生在前 0.3 秒，在色彩被大脑处理之前。你需要的不是天赋，而是一个系统：**去色诊断 → 三区四模板选型 → 九级明度阶定档 → 灰度先于色彩 → 三点光照执行 → 去色验证**。每画一个场景，先问自己：去色后，我的画面还能工作吗？如果不能，加什么颜色都没用。

**记住五件事**：
1. **明度是骨架，色彩是皮肤**——骨架歪了，皮肤再美也没用（Gurney）。
2. **去色检验 = 排除变量 debug**——禁用色彩测明度骨架，程序员天然的美术 debug 模式。
3. **三区四模板定基调，16 格估算法自检**——选模板就是选情绪，先选 preset 再画。
4. **灰度→上色 = 先写 HTML 再加 CSS**——结构（明度）对了，样式（色彩）怎么改都不会塌。
5. **九级明度阶 + 眯眼测试 = 量化自检**——把 0–255 量化成 9 档，眯眼做低通滤波看结构是否还在。

**如果只记住一句话：** 先在灰度下把骨架建对，再叠颜色——明度是 HTML，色彩是 CSS，结构没塌之前别动样式。

**上手行动**：今晚打开练手02 的 `character-v02-volume.aseprite`，按 L1 用明度思维重打一次光，存成 `character-v03-value.aseprite`。完成后按一次 Ctrl+Shift+U——如果去色后角色比练手02 那版更清晰、焦点更明确，你的明度直觉就长了一层。
