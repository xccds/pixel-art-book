# 练手08 像素专属技法：抗锯齿、Banding 与透视

### 8.0 这一章解决什么问题

练手07 你给贯穿角色定了动作线、摆了构图场景，存成 `character-v06-pose.aseprite`。线条、形状、空间、明度、色彩、质感、构图——八概念你走过了七个。但像素艺术还有一组问题不属于任何一个"概念"，只属于像素这个媒介本身：为什么低分辨率曲线总有锯齿？为什么有些边缘看起来像"梯子"？为什么不同游戏要用不同的"视角网格"？

**这一章不在八概念框架里——它是像素艺术独有的工程问题。** 八概念是任何视觉艺术都有的通用维度；抗锯齿（Anti-Aliasing）、Banding、游戏透视这三件事，只在"离散像素网格"这个约束下才出现。油画不需要抗锯齿，照片不会有 banding，雕塑不选透视投影。它们是像素这个离散媒介的"专属 bug 与专属 workaround"。

这一章把三件事拆开：**8.1 抗锯齿**——什么时候用、怎么用、用多少、用过头的危害；**8.2 Banding**——像素艺术公认的头号技术错误，怎么识别、怎么修；**8.3 游戏透视**——侧视/俯视/等距/二测/斜视，以及那个著名的"塞尔达透视问题"。

### 8.1 抗锯齿（Anti-Aliasing / AA）

#### 8.1.1 程序员视角：AA 到底是什么

练手01 讲过锯齿（jaggies）的成因——曲线段长跳变，人眼感知为断裂。消除锯齿有两个层次：第一层是练手01 的"阶梯原理"，让段长渐变不跳变（结构性修复）；第二层就是本章的**抗锯齿**——当阶梯原理已做到极限、曲线仍因分辨率不够而显"硬"时，在台阶转角处放一两个**中间明度的像素**，骗眼睛感知"平滑过渡"。

> **程序员类比 #1（AA 本质）：** 抗锯齿 = **亚像素插值**，像 CSS 的 subpixel rendering 或图像的 bilinear 缩放——你不是真的在亚像素位置画了东西，而是在硬边缘旁放一个中间色像素，让视觉系统自己做插值。浏览器在非整数 DPI 屏幕上渲染 1px 边框时会用灰度抗锯齿，逻辑完全一样：一个像素的位置无法精确表示那条线，就用相邻像素的亮度去"投票"。

关键区分：**AA 不是把线画得更直，是在已合规的线上"柔化"转角。** 如果你的线本身违反阶梯原理（段长 5→1→5 跳变），AA 救不了你——先修斜率，再谈 AA。

#### 8.1.2 什么时候用 AA，什么时候不用

AA 是工具，不是义务。用错地方比不用更糟。

| 适合用 AA | 原因 | 不适合用 AA | 原因 |
|----------|------|------------|------|
| 低分辨率曲线（<64px） | 阶梯感最刺眼，AA 收益最大 | 1-bit 艺术（黑白双色） | 没有中间色可用 |
| 高对比边缘（黑轮廓/亮背景） | 硬边突兀，AA 柔化 | 极小精灵（<16×16） | AA 像素占比过高，画面发糊 |
| 大尺寸曲线 | 台阶长，AA 能铺开过渡 | 刻意复古像素风 | 锯齿本身就是风格语言 |
| 角色面部/眼睛区域 | 细节区需要柔和 | 低对比度邻接 | 颜色接近时锯齿本就不明显 |

> **经验法则：** 先用练手01 的阶梯原理把线画合规，再看 100% 缩放下是否还"硬"。还硬才上 AA。AA 是锦上添花，不是雪中送炭。

#### 8.1.3 AA 的应用方法

三条核心规则：

**① 放多少 AA 像素？** 大约是台阶长度的一半。一个 4 像素宽的台阶，转角处放 2 个 AA 像素。**太少比太多好**——少了还能补，多了就糊。

**② 用几种过渡色？** 入门用 1 种（在线条色和背景色之间取一个中间值）；熟练后用 2 种（两个梯度）；最多 3 种。超过 3 种你就是在画矢量图不是像素画。

**③ 放在哪些位置？** 只放"内角"（线条弯曲的内侧），不碰"外角"（外侧保持干净锐利）。

![抗锯齿原则——中间色像素的放置](images/练手/练手08-fig01-抗锯齿原则.svg)

*图 练手08.1：左——合规但生硬的阶梯曲线。右——在内角转角处放中间色 AA 像素，视觉感知为平滑过渡。注意 AA 只在内侧，外侧保持锐利。*

#### 8.1.4 用 AA 控制线条粗细

AA 的一个副作用可变成工具：在线条一侧放**深色** AA 像素，线条视觉上变**粗**；放**浅色** AA 像素，线条视觉上变**细**。这让你在不改变实际像素数的前提下微调线条的"视觉重量"。

> **程序员类比 #2（线粗控制）：** 这像 CSS 的 `font-weight`——字符的实际像素没变，但渲染引擎通过亚像素灰度调节让 400 看起来比 700 细。你不是真的加粗了笔画，是调了边缘的明度投票。嘴巴、眼睛这些需要精细控制粗细的小区域，用这个技巧比真的换笔刷大小更可控。

#### 8.1.5 AA 过度的危害

AA 好用，但过度使用是初学者最常见的错误之一。三个危害：

- **模糊感**：太多 AA 像素让画面看起来像被高斯模糊过的矢量图，丧失像素艺术特有的"颗粒锐利感"。
- **颜色浪费**：每多一种过渡色就多吃一个调色板槽位——有限调色板下这是真金白银的代价。
- **身份丢失**：像素艺术的视觉身份建立在"你能看见格子"上；过度 AA 把格子藏起来，等于放弃这个媒介的独特性。

![过度 AA 对比——锐利像素 vs 模糊过度](images/练手/练手08-fig02-过度AA对比.svg)

*图 练手08.2：左——适量 AA，像素身份清晰、转角柔和。右——过度 AA，过渡色太多，看起来像缩放的矢量图，失去"像素画"的质感。*

> **程序员类比 #3（过度 AA）：** 过度 AA = **把所有强类型全改成 `any`**。TypeScript 的价值在于类型让错误在编译期暴露；像素艺术的价值在于离散网格让边缘清晰可读。你把每个边缘都"柔化"成多级过渡，等于把每个变量都标成 `any`——类型系统形同虚设，像素艺术的"类型身份"也模糊了。**诊断方法：缩到 100% 看。如果 100% 下像模糊的 PNG，你 AA 过度了——删掉一些过渡像素，让格子重新露出来。**

### 8.2 Banding —— 像素艺术的"死敌"

#### 8.2.1 程序员视角：Banding 是什么

如果说锯齿是像素画的"高频噪声"，那 **Banding** 就是"低频冗余"——两排（或多排）像素**完全平行**地紧贴在一起，沿一条直线形成整齐的阶梯条纹。人眼不把它读成"一条边"，而是读成"一架梯子"。

> **程序员类比 #4（Banding 本质）：** Banding = **规律性冗余行**，像代码里重复的 boilerplate——lint 该警告。想象你看到三行几乎一模一样的代码紧挨着，只有变量名差一个字符：你的第一反应不是"这三行在做不同的事"，而是"这该抽成一个函数"。眼睛对 Banding 的反应一样——它不读成"一条边"，它读成"三排重复的结构"，然后开始找"为什么重复"——注意力被引到了本不该被注意的地方。

#### 8.2.2 Banding 的危害

Banding 之所以是"死敌"，因为它同时制造四个问题：

1. **曲线方块化**：本该平滑的曲线，因平行阶梯而看起来像折线。
2. **线条显粗**：两排平行像素叠在一起，视觉上等于线条加粗了一倍——但你没画那么粗。
3. **轮廓过度模糊**：如果 banding 发生在 AA 层，过渡区域变成糊状带。
4. **枕头着色（Pillow Shading）**：阴影沿轮廓均匀包裹，没有明确光源方向——这是新手光影最典型的症状，根源往往是 banding。

#### 8.2.3 如何识别 Banding

诊断方法很直接：**找"完全平行的像素行"**。具体操作：

1. 沿着一条边缘逐行看，如果连续两三行的像素在相同的相对位置形成完全平行的阶梯——那就是 banding。
2. 特别检查"内描边 + 外轮廓"重叠的区域——两排深色像素紧贴是最常见的 banding 温床。
3. 检查 AA 层——如果 AA 像素和轮廓像素也形成平行阶梯，等于在 banding 上叠 banding。

![Banding 示例——平行阶梯条纹](images/练手/练手08-fig03-banding示例.svg)

*图 练手08.3：边缘处两排像素完全平行紧贴，形成"梯子"条纹。眼睛读到的不是"一条边"，而是"三排重复结构"——注意力被引到错误的地方。*

#### 8.2.4 三种修复方法

Banding 的修复只有一条核心逻辑：**打破平行**。三个具体手法：

**方法 ① 移除一个像素。** 从平行排列中删掉一个像素，让那一排断开——平行结构立刻瓦解。

**方法 ② 添加一个像素。** 在其中一排多塞一个像素，改变对齐方式——两排不再严格平行。

**方法 ③ 换更深的 AA 色。** 如果 banding 出现在 AA 层，把其中一排换成更深的过渡色，让平行区域产生层次差——"重复行"变成"渐变行"。

> **程序员类比 #5（修复 Banding）：** 修 Banding = **打破规律性冗余**，对应代码里发现重复行后的三种处理：① 删掉冗余行（移除像素）；② 把其中一行改成有意义的差异（添加像素改变对齐）；③ 用更精确的类型/参数让"看似重复"的行产生实质区别（换更深的 AA 色造层次）。三者的共同点：让每一行都"有理由在那里"，而不是机械复制。

![Banding 修复——三种手法](images/练手/练手08-fig04-banding修复.svg)

*图 练手08.4：三种修复手法对应三种"打破平行"的方式。核心逻辑只有一个——别让两排像素完全对齐。*

### 8.3 游戏透视（Game Perspectives）

#### 8.3.1 程序员视角：透视是什么

游戏里的"透视"不是美术课讲的那种有灭点的线性透视——那种要求远小近大，计算量大且不适合 tile 化。游戏几乎全用**伪 3D 投影**：物体不会因距离远而变小（无透视缩放），但通过固定的"视角网格"暗示三维深度。

> **程序员类比 #6（透视 = 数据结构）：** 选透视 = **选数据结构**。
> - **侧视 = 一维数组**：角色在一条线上左右移动，`x[]`，最简单。
> - **俯视 = 二维数组**：角色在网格上移动，`grid[x][y]`，Tile 天然适配。
> - **等距 = 带变换矩阵的二维数组**：还是 `grid[x][y]`，但渲染时乘一个 2:1 坐标变换矩阵，把正方形网格画成菱形。
> - **真透视 = 带深度缩放的投影**：`project(x,y,z)`，每个点都要除以 z——计算量大，像素画不常用。
>
> 你选的透视决定了关卡数据的存储结构、碰撞检测的维度、动画需要画几个方向。**选错透视 = 选错数据结构 = 后面所有系统都要为这个错误买单。**

#### 8.3.2 正交投影类型

大多数 2D 游戏用正交投影（Orthographic Projection）——无透视缩放，物体远近一样大。有四种基本类型：侧视、俯视、顶视、等距。加上二测和斜视，共六种常用视角。

![透视类型对比——五种视角](images/练手/练手08-fig05-透视类型对比.svg)

*图 练手08.5：同一个立方体在五种透视下的呈现。选择透视不只是"好看不好看"——它决定了你的关卡怎么搭、动画画几个方向、Tile 怎么切。*

#### 8.3.3 侧视角（Side View）

最经典的 2D 游戏视角，平台跳跃游戏（《超级马里奥》《蔚蓝》《空洞骑士》）的标配。角色在单一二维平面上移动，路径就是一条线。三种变体：剖面图（正对前方）、俯斜角（略高于地面）、俯视角度（从上方看的侧视）。无论哪种，角色移动路径始终在一维上——这也是为什么侧视的动画最简单（只需画左右两个方向）。

> **程序员类比 #7（侧视）：** 侧视 = **一维数组** `positions: number[]`。碰撞检测只查 `x` 轴，动画只需 `facing: 'left' | 'right'`。这是为什么平台游戏在独立开发中最常见——数据结构最简，工程成本最低。

#### 8.3.4 俯视角（Top-Down）

RPG 和动作游戏的常见视角（《星露谷物语》《暗黑破坏神》《以撒的结合》）。网格由正方形 Tile 构成，适合构建可探索的世界。根据相机高度分三档：正面 > 顶部（接近侧视）、正面 ≈ 顶部（经典 45° 俯视）、正面 = 顶部（完全垂直俯视）。

> **程序员类比 #8（俯视）：** 俯视 = **二维数组** `tilemap: Tile[x][y]`。碰撞检测查 `x` 和 `y` 两个轴，动画需要四个或八个方向。Tile 系统天然适配——每个 Tile 就是一个数组元素。

#### 8.3.5 "塞尔达透视问题"

俯视角有一个著名的坑：《塞尔达传说》系列中，**角色以 45° 俯视角度绘制**（你能看到角色的正面和头顶），**但房间以 90° 完全垂直俯视呈现**。两种视角混在一起，当角色靠近南墙时，视觉上看起来像是"躺在地板上"——因为角色有"正面"而地板没有深度。

![塞尔达透视问题——角色视角与场景视角不匹配](images/练手/练手08-fig07-塞尔达透视问题.svg)

*图 练手08.7：角色用 45° 俯视绘制（有正面），房间用 90° 垂直俯视绘制（无正面）。两者叠在一起，角色靠近南墙时看起来像"躺在地上"。*

> **程序员类比 #9（塞尔达问题）：** 塞尔达透视问题 = **坐标系不匹配 / 类型不一致**。你的角色数据是 `{x, y, facing}` 带"朝向"信息，但你的场景数据是 `{x, y}` 纯网格——两套坐标系混在一起渲染就会出现"类型错误"。解法只有两个：要么角色也改成纯俯视（牺牲角色表现力），要么场景也改成 45° 俯视（牺牲 Tile 管理简单性）。你不能两边都占——选了 45° 角色就得接受 45° 场景的复杂度。

#### 8.3.6 等距视角（Isometric）

最受欢迎的伪 3D 视角（《暗黑破坏神》《最终幻想 战略版》）。使用菱形网格，能看到物体的顶面、左面、右面三个面。所有轴等比例，看起来像"等距投影"。

**2:1 阶梯规则：** 真正的等距投影角度是 30°，但像素网格上无法画精确的 30° 线。像素艺术用 **2:1 比例**近似——每 2 个水平像素对应 1 个垂直像素，实际角度约 26.5°。这是整数网格上能近似等距的**最简整数比**。

![等距 2:1 网格——阶梯规则](images/练手/练手08-fig06-等距2比1网格.svg)

*图 练手08.6：等距菱形网格由 2:1 阶梯线构成。每条斜线水平走 2 格、垂直走 1 格，近似 26.5°（而非精确 30°）。这是像素网格上最简的等距近似。*

> **程序员类比 #10（2:1 等距）：** 等距 2:1 = **一个固定的坐标变换矩阵**。把正方形网格 `[x, y]` 乘以矩阵变换，就旋转成菱形等距视角。2:1 之所以是"最简整数比"，是因为它在像素网格上能用整数运算精确表达——就像 Bresenham 算法用整数画线一样，你不想在渲染管线里引入浮点数。30° 需要 `tan(30°) ≈ 0.577`，不是整数比；2:1 给你 `atan(1/2) ≈ 26.5°`，差 3.5° 但完全用整数可表达。

等距网格有两种用法：**自由形式**（不严格对齐网格，适合插画式像素画）和 **Tile 形式**（严格对齐菱形网格，适合游戏开发，每个菱形是一个 Tile）。

#### 8.3.7 45° 二测视角（Dimetric）

与等距（30°）不同，二测视角的两个侧面角度为 45°。较少见，适合场景中有高大建筑时——让建筑本身不会遮挡游戏玩法。它是一种"等距的变体"，轴比例不全部相等。

> **程序员类比 #11（二测 vs 等距）：** 二测 = 等距变换矩阵的**参数变体**。等距是三个轴等比缩放，二测是其中两个轴用不同比例——就像你把 `transform: scale(1, 0.5)` 改成 `scale(1, 0.7)`，投影矩阵参数变了，渲染角度就不同。选二测通常是因为等距下建筑太高会挡住后方——二测压扁了垂直轴，让镜头能看到更多。

#### 8.3.8 斜视视角（Oblique View）

正面是平的（90° 直面），侧面是斜的。强调垂直平面的表现，适合建筑内部或需要突出正面细节的场景。你在工程蓝图里见过的"斜二测投影"就是这种——正面画真实形状，深度方向画 45° 斜线。

#### 8.3.9 如何选择透视

四个决策因素：

| 因素 | 侧视 | 俯视 | 等距 |
|------|------|------|------|
| 游戏玩法 | 平台跳跃、横版射击 | RPG、动作、解谜 | 策略、模拟经营 |
| 像素数量 | 少（一维） | 中（二维方格） | 多（二维 + 变换） |
| 动画复杂度 | 低（2 方向） | 中（4-8 方向） | 高（8 方向 × 等距角度） |
| Tile 友好度 | 高（矩形） | 高（矩形） | 中（菱形需专用 Tile） |

**核心原则：游戏玩法优先。** 透视服务于游戏设计，不是反过来。你不会因为"等距好看"就给一个平台跳跃游戏用等距——那等于给一维数据硬套二维变换，工程上全是不必要的复杂度。

### 8.4 练习

本章练习逐级递进。L1 和 L3 继续贯穿角色项目，L2 是独立的 Banding 专项。

> **贯穿式角色项目（续）：** 练手07 你存了 `character-v06-pose.aseprite`。**L1 给它做 AA 打磨 + Banding 清理**，存成 `character-v07-aa.aseprite`；**L3 为它的游戏选定透视并画等距版本**，存成 `character-v07-iso.aseprite`。这个角色接下来交给**制作02（像素角色工作流）**收口。

#### L1 · 给贯穿角色做 AA 打磨 + Banding 清理（约 45-90 分钟）

**目标**：把练手07 的 `character-v06-pose.aseprite` 从"线条合规但生硬"升级为"AA 恰当、无 Banding"——做一次上线前的像素质检。

**你需要**：练手07 的 `character-v06-pose.aseprite`、Aseprite。

**步骤**：
1. 复制 `character-v06-pose.aseprite` 为 `character-v07-aa.aseprite`。
2. **AA 打磨**：放大到 400%，沿角色外轮廓和内部曲线逐段检查。在台阶转角的**内角**放 1 种中间色 AA 像素，数量约为台阶长度的一半。先从面部和手部这些高注意区域开始。
3. **100% 验证**：缩回到 100%，看转角是否比之前柔和。如果看起来模糊——你 AA 过度了，删掉一些。
4. **Banding 巡查**：逐段检查内描边和外轮廓重叠处——找有没有两排像素完全平行紧贴。发现一处，用 8.2.4 的三种修复手法之一打破平行。
5. **去色复查**：按 Ctrl+Shift+U 去色，灰度下 banding 更易被发现（颜色对比被去掉后，平行结构更突出）。
6. 保存为 `character-v07-aa.aseprite`。

**合格标准**：
- [ ] 外轮廓曲线转角处有 1 种中间色 AA 像素（不是 0 种也不是 3 种以上）。
- [ ] 100% 缩放下转角比打磨前柔和，且不模糊。
- [ ] 逐段巡查后无两排像素完全平行紧贴的 Banding。
- [ ] 文件存为 `character-v07-aa.aseprite`，准备交给 L3 和制作02。

#### L2 · Banding 诊断与修复专项（约 20-40 分钟）

**目标**：独立于角色项目，纯练 Banding 的"发现—修复"技术。

**你需要**：Aseprite，新建 48×48 画布。

**步骤**：
1. 画一个简单的实心圆（直径约 32px），用单色填充，外加一圈深色描边。
2. 逐段检查圆的边缘——描边和填充之间、描边和外轮廓之间，找平行阶梯。
3. 标记所有 Banding 位置（新建一个红色图层在上面画框）。
4. 逐一用三种修复手法处理：先试"移除一个像素"，不行再试"添加一个像素"，最后试"换更深的 AA 色"。
5. 修复后缩到 100%，对比修复前后——"梯子感"应该消失。

**合格标准**：
- [ ] 至少找出 3 处 Banding 并标记。
- [ ] 每处都用了三种手法之一修复，且能说出用了哪种。
- [ ] 100% 缩放下"梯子感"明显减弱或消失。
- [ ] 修复没有引入新的锯齿（没有破坏练手01 的阶梯原理）。

#### L3 · 为角色游戏选透视 + 等距重绘（约 60-120 分钟）

**目标**：给贯穿角色"所在的那个游戏"选定一种透视，并画一个等距版本的简化角色。

**你需要**：L1 的 `character-v07-aa.aseprite`、Aseprite、纸笔。

**步骤**：
1. **选透视**：在纸上写下你贯穿角色"所在的游戏"是什么类型——平台跳跃？俯视 RPG？等距策略？根据 8.3.9 的四因素表选定一种透视。写下一句理由（"选俯视因为我的游戏是地牢探索，需要 360° 移动"）。
2. **等距网格**：在 Aseprite 新建 64×64 画布，用 2:1 阶梯规则画一个等距菱形网格（每条斜线水平 2、垂直 1）。参照图 练手08.6。
3. **等距角色**：在等距网格上画一个简化版你的贯穿角色——不需要完整细节，只要能看出它在等距视角下的三个面（顶/左/右）。存成 `character-v07-iso.aseprite`。
4. **如果你选的不是等距**：也要画这个等距练习（练手 2:1 规则），但额外在纸上画一个你选定透视的角色速写。
5. 对比你的原版透视和等距版——体会"选透视 = 选数据结构"：等距版是不是明显更费像素、更费时间？

**合格标准**：
- [ ] 写下了游戏类型 + 选定透视 + 一句理由。
- [ ] 等距网格符合 2:1 阶梯规则（逐行数过水平/垂直像素比）。
- [ ] 等距角色能看出三个面，不只是侧视的复制粘贴。
- [ ] 文件存为 `character-v07-iso.aseprite`。
- [ ] 能用一句话说出原版透视和等距版在"工程成本"上的差异。

### 8.5 常见错误与诊断

**错误 1：AA 用太多 → 画面发糊。** 缩到 100% 看着像缩放的矢量图，格子全被过渡色淹没。**原因**：把 AA 当"越多越好"，每段都铺 2-3 种过渡色。**修复**：删到只剩 1 种过渡色，只保留台阶最长处的 AA。记住"太少比太多好"。**过度 AA = 把强类型全改成 `any`——类型身份模糊了。**

**错误 2：忽视 Banding → 边缘像梯子。** 你画完觉得"还行"，但总觉得哪里不对——边缘有一排整齐的条纹让画面"廉价"。**原因**：内描边 + 外轮廓叠加时两排深色像素完全平行，你没有检查。**修复**：去色后逐段巡查，找到平行行就打破。Banding 是"死敌"不是因为它难修，是因为它难发现——你得训练眼睛去找"完全平行"。

**错误 3：AA 放在外角 → 边缘溃散。** 你在曲线外侧也放了 AA 像素，结果外轮廓不再锐利，形状和背景糊在一起。**原因**：没区分内角外角——AA 只该放内侧。**修复**：删掉所有外角 AA，外轮廓保持 1px 锐利。AA 是"内侧柔化"，不是"外侧晕染"。

**错误 4：透视与玩法不匹配 → 工程灾难。** 你因为"等距好看"给一个平台跳跃游戏用了等距视角，结果碰撞检测、关卡设计、动画方向全都要重做。**原因**：先选透视再想玩法，本末倒置。**修复**：**游戏玩法优先**。侧视给平台跳跃，俯视给 RPG，等距给策略——数据结构服务算法，不是反过来。**选错透视 = 选错数据结构 = 后面所有系统买单。**

**错误 5：塞尔达透视混用 → 角色像躺在地上。** 你用 45° 角色配 90° 场景，角色靠墙时视觉崩坏。**原因**：没意识到角色和场景是两套坐标系。**修复**：统一视角——要么角色也改成纯俯视，要么场景也改成 45°。不能两边都占。

### 8.6 本章小结

这一章不在八概念框架里——它是像素这个离散媒介的**专属工程问题**。八概念是任何视觉艺术都有的通用维度；抗锯齿、Banding、透视只在"整数像素网格"这个约束下才出现。

**记住五件事**：
1. **AA = 亚像素插值**——在合规线条的内角转角放中间色像素，骗眼睛感知平滑。太少比太多好；过度 AA = 把强类型改成 `any`，模糊了像素身份。
2. **Banding = 规律性冗余行**——两排像素完全平行紧贴，眼睛读成"梯子"而不是"边"。它是像素艺术的头号技术错误，但难发现不难修。
3. **修 Banding 只有一条逻辑：打破平行**——移除像素 / 添加像素 / 换更深 AA 色，三选一。
4. **等距 2:1 = 固定坐标变换矩阵**——用最简整数比近似 30° 等距，实际 26.5°，但完全可用整数运算表达。
5. **选透视 = 选数据结构**——侧视=一维数组，俯视=二维数组，等距=带变换的二维数组。游戏玩法优先，透视服务玩法。

**如果只记住一句话：** 像素艺术的三件专属工程——AA 是亚像素插值（少比多好），Banding 是冗余行（打破平行），透视是数据结构（玩法优先）。

**上手行动**：今晚打开练手07 的 `character-v06-pose.aseprite`，按 L1 给它做一次 AA 打磨 + Banding 清理，存成 `character-v07-aa.aseprite`。再按 L3 为它的游戏选一种透视，画一个 2:1 等距版本存成 `character-v07-iso.aseprite`。完成后缩到 100%——如果转角柔和但不模糊、边缘没有梯子、等距版本能看出三个面，你的像素工艺就过了一道质检关。这个角色将在制作02 收口。
