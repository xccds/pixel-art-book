# 继续01 GameJam：48小时的艺术极限测试

### 1.0 这一章解决什么问题

制作09 收尾时，你走完了第四部"制作"的全部 9 章，有了"看 → 做 → 整"的完整视觉系统。第五部"继续"不再教你新技术——它问的是：**你能不能在极端压力下把这套系统用出来？** Game Jam（游戏开发马拉松）——48 小时硬约束下从零做出一款可玩游戏——对独立开发者不是比赛，是高压视觉训练场。

2002 年，印第安纳州。一个叫 Mark Essen 的 19 岁大学生和两个朋友决定花一个周末做一款游戏。他们用 Game Maker 拼出了一个小人从僵尸堆里逃命的玩意儿，画面粗糙到主人公只是一团灰色方块，僵尸是红色方块，墙壁是黑色方块。但那款叫做《Flywrench》原型的游戏在当天的小型 Game Jam 上拿了第一名。不是因为好看——是因为所有人一眼就看懂了该干什么、角色往哪跑、谁在追你。

四年后 Mark Essen 以《Nidhogg》再度出山，那款击剑格斗游戏的画面风格——单色剪影加大面积纯色背景——其实就是从那个 48 小时块块游戏里长出来的。**他不是先学会画画再去做《Nidhogg》的。他是用 48 小时的硬约束逼自己找到了一个"不需要会画画也能让画面成立"的解法。**

这就是 Game Jam 对独立开发者的真正价值。不是练技术。是练一种更根本的东西——**在极端时间压力下，区分"必须画"和"不画也行"。**

对程序员来说，这相当于你第一次参加 hackathon 时被逼着砍掉 80% 的功能需求，只留一个可演示的 demo。你发现原来用户不需要那个花了两周设计的权限系统，一个简单的 token 就够了。Game Jam 给你的视觉能力训练，是在 48 小时内反复做这个判断：这个画面元素，玩家真的会注意到吗？如果不会，不画。

本章教你如何把 Game Jam 变成一次集中的视觉极限测试。你不只是去写代码。你要去练习"48 小时内让游戏看起来像成品"的能力。

> **本章核心承诺：** 读完本章，你会在下一次 Game Jam 中用 48 小时的时间表规划视觉工作，而不是在最后 6 小时才惊觉所有素材需要推翻重来。你会掌握五种低门槛但成立的美术风格选择标准（像素作为本书读者的首选）、一份"看起来像成品"的 5 要素检查清单、以及一套赛后视觉复盘模板——把每一次 Game Jam 变成一次集中的视觉极限训练。

### 1.1 为美术准备的 Game Jam 策略

#### 不是做最好的，是做"看起来像成品"的

大多数程序员参加 Game Jam 的心态是："我程序写得好，画面随便贴几个方块就行，反正拼的是玩法。"这种想法本身没错，但如果你的长期目标是成为一个能独立完成商业级作品的全栈开发者，那这个心态就是**浪费了一次完美的高压视觉训练机会**。

有一个反直觉的事实：**在 48 小时的时间约束下，追求"正确"的视觉决策往往比追求"好看"的视觉决策更有效。**

什么叫"正确的视觉决策"？

- 选择一种风格，然后**全游戏严格统一它**。哪怕这个风格是"纯色块 + 粗描边"，只要每个画面元素都遵守，看起来就是刻意的、有设计的——而不是你不会画。
- 限制你的调色板。48 小时里，4-6 种颜色足够你用。多出来的颜色都会变成"需要决策的成本"。
- 把所有 UI 元素（血条、分数、按钮）在最后一口气统一做。不要在画角色的同时随手弄个按钮——统一制作 UI 的结果是它们看起来属于同一个游戏。

> **程序员类比**：这就像你的代码仓库设置了 eslint 和 prettier。在 48 小时的 hackathon 里你不可能写出完美的架构，但你至少可以保证缩进一致、命名规范、没有 magic number。视觉的"linting"就是风格统一、配色统一、UI 统一。做到这三件事，你的游戏看起来就比 90% 的 jam 作品更专业。

#### 选一种"省力但成立"的美术风格

以下是在 Game Jam 中行之有效的几种低门槛风格，按"出效果 / 投入时间"比率（ROI）排序。**注意：这是通用 ROI 排名——对你（本书读者）来说，首选是第 3 项像素，原因见后。**

**1. 几何极简（Geometric Minimalism）**
纯色块 + 圆角矩形。所有角色是几何体组合，背景是大色块。参考：《Thomas Was Alone》。一张角色图需要 5 分钟——两个矩形加个圆。你不需要会画人体，只需要会分辨什么形状组合"看起来像一个人"。

**2. 单色剪影（Monochrome Silhouette）**
黑色剪影 + 纯色背景。参考：《Limbo》《Nidhogg》。优势是彻底回避了内部细节——你不会画五官？剪影不需要五官。你不会画衣服褶皱？剪影只需要外轮廓。外轮廓好不好看？画 3 个版本，选最有辨识度的。

**3. 像素艺术（Pixel Art）——本书读者的首选**
16×16 或 32×32 的精灵。参考：《Celeste》《Downwell》。每个角色就是一个极小的画布，在这个尺寸下"丑"是模糊的——你画歪了一像素，整个画面缩小后根本看不出来。而且 Aseprite 这类工具极其适合快速迭代。

> **为什么像素是本书读者的首选——即使它通用 ROI 排第三？** 两个原因：第一，**它是你已经在制作01-05 训练过的技能，零学习成本**——几何极简和剪影你都得临时摸一套新工作流，像素你已经会了；第二，**Aseprite → Godot 是你已经拥有的最短管线**（制作01 锁定的工具栈），从 `.aseprite` 到屏幕零中间格式、零格式转换 debug。48 小时里没有"学一套新工具链"的预算——用你已经熟的那套。其他四种风格作为"如果游戏类型不允许像素"的备选。

**4. 手绘扫描（Hand-Drawn Scan）**
纸笔画线，手机拍照，进电脑简单处理后直接使用。参考：《Baba Is You》的早期原型。优势是跳过了一切数字绘画工具的学习曲线。你用了一辈子的铅笔，你其实**已经**会用它了。适合你想在 48 小时里刻意跳出像素练手感的场景。

**5. UI-only（纯界面）**
如果你的游戏类型允许（文字冒险、卡牌、策略），完全放弃角色和场景，用纯 UI + 排版做画面。好的字体选择 + 间距 + 配色可以让一个纯界面的游戏看起来极其高级。

选择风格的核心标准：**48 小时内，你能重复画出第 50 张同风格素材，且第 50 张和第 1 张看起来是同一个人画的。**

这就是为什么像素和几何极简这么受欢迎——不是大家怀旧，是在极端时间压力下，这些风格**允许你不一致**。像素的"不一致"被低分辨率吸收了。几何极简的"不一致"被形状的数学精确性约束了。

#### "看起来像成品"的 5 要素检查清单

在提交前最后 2 小时，拿着这张清单过一遍你的游戏：

1. **所有文字用同一字体吗？** → 如果用了 3 种字体，选一种，全换。
2. **所有按钮风格一致吗？** → 如果有的带边框有的不带，统一。
3. **调色板有超过 8 种颜色吗？** → 挑最常用的 5 种，把其他的替换掉。
4. **有"裸奔"的 UI 元素吗？** → 纯白底黑字的按钮就是裸奔的，给它加个底色。
5. **所有画面的明暗调子一致吗？** → 把游戏截图转黑白，如果有的画面很亮有的很暗——校色。

一个常见的程序员陷阱：你觉得"再加一个粒子效果会让画面更好"，于是最后 1 小时加了个粒子系统。然后因为时间不够，粒子效果的配色和你整体调色板冲突。**加分的特效变成了扣分的视觉 bug。** 在最后阶段，删除比添加更能提高"成品感"。

### 1.2 48 小时视觉冲刺时间表

![48小时视觉冲刺时间线](images/继续/继续01-fig01-48hour-sprint.svg)

*图 继续01.1：48 小时视觉冲刺时间线（48-Hour Visual Sprint Timeline）——五个关键节点把 48 小时切成可执行的节奏：第 1 小时锁风格、第 3 小时出角色、第 8 小时场景可跑、第 24 小时核心动画、第 44 小时 UI 打磨。核心原则：不是做最好的，是做"看起来像成品"的——完成度 > 精细度。*

这不是一个你需要严格按分钟执行的战术计划。这是一个**节奏模板**——它为你的 48 小时提供了一个视觉工作的"骨架"，让你不至于在最后 6 小时才惊觉自己画的所有东西都需要推翻重来。

#### 第 1 小时：风格锁定（Style Lock）

**产出**：一张风格指南——不超过 1 页。

坐在电脑前，打开 Google Images 或 Pinterest，搜 3 个关键词——游戏类型截图（如 "top-down pixel game indoor"）、颜色调板（如 "game palette 4 colors"）、情绪（如 "cozy game UI"）。从搜索结果中选最多 5 张参考图，扔到一张画布里。然后做三件事：
1. 从参考图中提取主色 3-5 种，写在画布旁边
2. 写下你的风格描述，用一句话（例如："深蓝主色调的俯视像素风，角色 2 头身，白色描边"）
3. 确认你的画布尺寸 / 精灵尺寸——这是和程序对接的关键参数，之后不再更改

> **这一页风格指南就是风格04《视觉风格文档》的 GameJam 1 页精简模板**——风格04 给出完整项目的三档模板（GameJam 1 页 / 小项目 1 页 Notion / 完整 wiki），这里是其中最快的那档。先用它，赛后想升级到完整版回风格04。

**为什么 1 小时？** 因为超过 1 小时你就在拖延了。完美风格不存在。存在的只有"还行并且我能 48 小时执行到底"的风格。**在 Game Jam 中，一个"还行"的决定现在做出，好过一个"完美"的决定 3 小时后做出。**

#### 第 3 小时：角色就绪（Character Ready）

**产出**：主角的站立姿态 ×1 个方向、至少一个其他关键帧（跳跃 / 攻击 / 受伤任选一）。

角色是玩家在这个游戏世界里唯一拥有的东西。如果一个游戏只有 5 张图，主角的图应该是其中 3 张。所以先把角色立起来。

技术要点：
- **先画剪影，再填内部**。用纯黑色画外轮廓，不断调整到"光看影子就能认出是什么"。这个来自动画业的原则在 Game Jam 中尤其好用——因为如果你的剪影不好看，填再多细节也不会好看。
- **站在场景中画**。不要在一个空白的透明背景上画角色。把场景（哪怕只是一个色块背景）放在角色图层下面画。这就是为什么第 1 小时你要先定调色板——角色必须在场景色中成立。
- **把角色的色板和场景色板合在一起，检查是否有颜色"撞"了**——如果角色的主色和背景主色接近，玩家就看不清角色。没有比一个看不清的主角更糟糕的游戏体验了。

> **像素角色的完整 10 步工作流见制作02《像素角色工作流》**——这里是它的 48 小时压缩版。一个节省时间的技巧：如果你做的是俯视 / 侧视游戏，角色的"背面"和"正面"可以通过翻转 + 微调来解决。先画一个方向，另外三个方向能翻转的就翻转，需要重画的再重画。不要一开始就画四个方向。

#### 第 8 小时：场景跑起来（Scene Playable）

**产出**：至少一个完整的可玩场景——角色可以在上面走、跳、交互。

第 3-8 小时的 5 小时是 Game Jam 中最重要的阶段。这个阶段的目标不是"画好看的场景"，是**在真正的游戏画面中检验角色的视觉表现**。

很多 jam 作品的问题是：角色单独看不错，场景单独看也不错，但放到一起就互相打架。冲突通常出在三个方面：**比例**（角色太大淹没场景细节，或太小看不清动作）、**对比度**（角色亮度和背景太接近，融为一体）、**风格**（线宽不一致——角色 3px 描边，场景 1px 描边）。

这 5 小时里至少要完成：
- 主场景背景 1 张（比如一个森林 / 房间 / 外太空）
- 可交互对象的 Tile / 瓦片规则确定（地板 / 墙壁 / 平台）
- 前景遮挡层（让角色"走进"树后面而不是"走在"树前面）

> **Tile 规则、3-Tile 过渡、三层叙事（前景 / 中景 / 远景）见制作03《环境与 Tile》**——GameJam 里你只需要 1 个场景，所以只取制作03 的最小子集：1 个背景 + 1 套地板墙壁瓦片 + 1 层前景遮挡。

> **程序员类比**：这就是你的"Hello World"——你最简可运行版本。在程序开发中，你绝不会先写完所有模块再拼起来测试，你会先做一个 MVP 看各部分能不能通信。场景的 MVP 就是：角色 + 背景 + 一个能走的地面——这三者在一起成立，游戏画面就成立了。

#### 第 24 小时：核心动画组（Core Animation Set）

**产出**：待机、行走、至少一个特殊动作的动画。

用 12-16 小时（中间要睡觉，假设你睡了 4-6 小时）来完成动画。

帧数建议：
- 4 帧动画是 Game Jam 的黄金标准。2 帧是"闪"（闪烁），6 帧是"顺"（顺滑）。4 帧是"像样"（看起来像那么回事）。
- 待机动画 3-4 帧，行走动画 4-6 帧，跳跃 2-3 帧。
- 不要做斜向动画。上下左右四个方向够用了。斜向动画 = 4 帧 × 4 个方向 = 16 张图，你没有那么多时间。

工具选择：**Aseprite 是像素动画的绝对王者**，洋葱皮（Onion Skinning）功能让你看清上一帧和下一帧的叠影；Piskel（免费在线）适合临时使用；或直接在引擎里做帧动画（Godot AnimatedSprite）跳过导出导入流程。

> **帧时序、走路循环、挤压拉伸见制作05《动画与特效》**——GameJam 里只取核心：待机 2 帧呼吸 + 行走 4 帧 + 一个特殊动作。制作05 的帧时序不等长原则（接触帧 150ms、关键帧 100ms、中间帧 60ms）即使在 48 小时里也值得手动设——只改帧时长不改像素，动画提升立即可见。

**"至少看起来能动"原则**：如果时间不够，先让角色能"走"。一个只能走的主角 + 能交互的场景 > 一个能做 20 种动作但场景一片空白的主角。玩家不会因为你没做攻击动画就认为你的游戏不好玩，但如果你连走路都卡顿，他们会立刻关掉。

#### 第 44 小时：UI 与一致性打磨（Polish & UI）

**产出**：统一 UI + 全局校色 + 可演示成品。

最后 4 小时（假设你有 4 小时余量交稿 / 调试），全部投入打磨：

**UI 统一**（1.5 小时）：所有按钮同一套样式（底色 / 描边 / 圆角 / 文字大小）；血条 / 能量条统一高度和风格；做一个字体层级决策（标题 / 正文 / 数字各多大）；菜单背景统一透明度和模糊度。

> **UI 8 步工作顺序、80-20 注意力分配见制作04《UI 设计》**——这里是它的 1.5 小时压缩版。GameJam 里你跳过完整 8 步，只做最关键的两步：统一所有按钮 / 条 / 字体（第 1 步布局），再做一次 3 秒扫视测试（第 8 步真人观察）。

**全局校色**（1 小时）：
- 把所有场景截图，放到一起看——有没有哪个场景"跳"出来（太亮 / 太暗 / 色温不对）
- 加一个全局后处理（post-processing）：微微的色偏、暗角（vignette）、或对比度调整，能让所有画面"统一经过同一滤镜"。最简单的做法是在画面上叠加一个半透明色块（10% 透明度），立刻统一色调。

**成品质感**（1.5 小时）：
- 加标题画面。标题画面是玩家对游戏的第一印象。不需要复杂，游戏名 + 背景图 + 一个 "Press Start" 的提示就够了
- 截图 3 张，作为 jam 提交页面的展示图。绝大多数 jam 评委只看截图就决定了要不要打开你的游戏
- 标题画面和这 3 张截图是你在 48 小时里 **ROI 最高**的视觉工作。因为它们是唯一能保证 90% 的人会看到的画面

### 1.3 Game Jam 视觉三大死法

**症状 1：风格漂移（Style Drift）。** 第 1 小时选了像素风，第 8 小时开始往里面加手绘质感，第 24 小时加了个矢量 UI 按钮。48 小时结束后你的游戏像三个不同的人分别画了三部分。
**原因：** 没有在第 1 小时锁死风格并写下来。口头说"我们做像素"不够——必须有一页写了调色板、线宽、画布尺寸的风格卡。
**修复：** 第 1 小时做的风格指南打印出来或固定到第二屏幕。每画一张新素材前看 3 秒。

**症状 2：最后 2 小时狂加特效。** 你觉得"再加一个粒子效果会让画面更好"，于是最后 1 小时加了个粒子系统。然后因为时间不够，粒子效果的配色和你整体调色板冲突。加分的特效变成了扣分的视觉 bug。
**原因：** 在时间压力下，"加东西"比"修东西"更有成就感——但你的目标是"看起来像成品"，不是"看起来有很多特效"。
**修复：** 严格按照 1.2 的时间表，最后 4 小时只做删减和统一，不做新功能。

**症状 3：跳过灰度检查。** 不同人画的素材（如果你是组队）或不同时间段画的素材（如果你是 solo），明度差异巨大却不自知。
**原因：** 你只在彩色模式下审素材，而彩色会掩盖明度差异。
**修复：** 第 44 小时，把所有截图转灰度并排在一起看。任何一张明显比其他的更亮或更暗——调。

### 1.4 赛后复盘——视觉层面的四个问题

![赛后复盘模板](images/继续/继续01-fig02-赛后复盘模板.svg)

*图 继续01.2：赛后复盘模板（Game Jam Postmortem）——视觉层面四象限评估：视觉一致性 / 动画流畅度 / UI 清晰度 / 配色统一性。每个象限三个自检问题 + 1-10 打分 + 最大问题 + 程序员类比。复盘不是打分，是提取一条可迁移的经验。*

Game Jam 结束后睡一觉，然后做一次视觉复盘。不是打分（你给自己打分会倾向于要么满分要么零分），是回答四组具体问题：

#### 视觉一致性（Visual Consistency）

把游戏所有截图平铺在屏幕上，问自己：
- 如果是一个不认识你的人看这些截图，Ta 会觉得是同一个游戏吗？还是像 3 个不同游戏的截图拼在一起？
- 有没有哪张素材让你觉得"这是我凌晨 3 点画的，看得出来"？如果看得出来——为什么？线变粗了？颜色偏离了？比例崩了？
- 你的素材库现在有多少张图？里面有没有你画了但完全没放进游戏的废稿？废稿多说明你在方向不明确时就开始画了。

**改进方向**：下一次 Jam，在开始画任何东西之前，先画一张"风格参考卡"——上面有你的调色板 + 线宽标准 + 角色比例尺。画每一张新素材时都放在这张卡旁边对照。这就是风格04 风格文档的 GameJam 1 页模板。

#### 动画流畅度（Animation Fluidity）

- 你的角色动画中，哪一帧和前后帧之间有明显的"跳跃"感？用 Aseprite 的洋葱皮回看，找到跳跃帧。
- 玩家操作到画面反馈之间有没有可感知的延迟？这个可能不是动画问题而是程序问题，但视觉上它表现为"不流畅"。
- 你用了 easing 吗？如果角色从 A 点移动到 B 点是匀速的，它就"不自然"。加一个最简单的 ease-in-out 能让动作顺畅 3 倍。

#### UI 清晰度（UI Clarity）

找一个人（不是你，没看过你做游戏的人），打开你的游戏，不说话，看 Ta 操作 30 秒。
- Ta 是否能不问你任何问题就完成基本操作？
- Ta 的视线在哪儿？是在看角色还是在看 UI？（如果是后者，你的 UI 太抢眼了）
- 有没有文字 Ta 看不清或读不懂？（"HP"可能只有玩家知道是血条，普通人不知道）

**改进方向**：用"**奶奶测试**"——如果你奶奶拿到这个游戏（假设她会用键盘），她能不能不靠任何说明就开始玩？不能的话，UI 该改。

#### 配色统一性（Color Unity）

- 把你的游戏截图转成灰度图。灰度图里最亮和最暗的区域有没有超过你想要的明暗范围？
- 有没有某个颜色只出现了一次？（比如全游戏只有一个绿色按钮，其他都是蓝色的）
- 你的角色在每一个场景里都能被清晰地分辨出来吗？把每个场景的截图都做一次"角色-背景对比度检查"。

### 1.5 赛后复盘模板

在 jam 结束后 48 小时内，花 20 分钟写一份简短的复盘。用下面的模板：

```
【项目名称】
【Jam 主题】（如果有的话）
【使用的美术工具】

【视觉一致性自评】
- 3 张代表截图放在一起，像同一个游戏吗？
- 最不一致的地方是哪里？原因是什么？

【动画流畅度自评】
- 最好的一个动画是什么？为什么好？
- 最差的一个动画是什么？为什么差？

【UI 清晰度自评】
- 有没有人用了你的游戏后问"这个按钮是干什么的"？
- 如果有，你怎么改？

【配色统一性自评】
- 实际用了多少种颜色？和你计划的差多少？
- 灰度图检查结果如何？

【如果再做一次，我会改什么？】
（只写 3 件——不是"把一切都画得更好"，是具体可执行的改动）
```

复盘的目的是**提取一条可迁移的经验**。不是"这次我的角色画得不好"，而是"我应该在画角色之前先确定线宽标准——因为这次所有角色的描边粗细都不一样，导致了视觉不一致。"

把这条经验写下来，放在你的开发笔记里。下一次 Jam 开始前翻出来看一遍。这样每一次 Game Jam 都建立在之前所有 Jam 的经验之上。

### 1.6 练习

**L1：模拟 48 小时风格锁定（15 分钟）**

不参加 Game Jam 也能练。选一个你未来想做（或正在做）的游戏概念，计时 15 分钟，完成以下三件事：
1. 在 Google Images 或 Pinterest 上找 3 张参考图，扔进一张画布里
2. 从参考图中提取 3-5 个主色，写在旁边（用取色器或肉眼判断）
3. 用一句话写风格描述（例如"深蓝主色调的俯视像素风，角色 2 头身，白色描边"）

**怎么检验：** 把这页风格指南发给一个朋友，问 Ta："只看这页，你觉得我要做一个什么样的游戏？"如果 Ta 的回答和你心里想的基本一致——你的风格指南是合格的。如果 Ta 说"我看不出来"——你的风格描述不够具体。

### 1.7 本章小结

Game Jam 不是比赛——是高压训练场。48 小时的硬约束逼你做出最诚实的视觉决策：什么必须画、什么可以不画、什么画了反而破坏一致性。三个关键：第 1 小时锁风格并写下来、每 8 小时灰度检查一次、最后 4 小时只删不加。赛后做一次四维度复盘——一致性、流畅度、清晰度、配色——总结出"一条下次可用的经验"。每一次 Game Jam 都建立在之前所有 Jam 的教训之上。对你——本书读者——来说还有一层红利：像素是首选不是因为它通用 ROI 最高，是因为它是你已经训练过的技能 + 你已经拥有的最短管线（Aseprite → Godot）——48 小时硬约束下"用你已经会的"永远赢过"临时学一套新的"。

> **如果只记住一句话：** Game Jam 不是比赛——是高压训练场；48 小时硬约束逼你做出最诚实的视觉决策——什么必须画、什么可以不画、什么画了反而破坏一致性。第 1 小时锁风格并写下来、最后 4 小时只删不加、赛后提取一条可迁移的经验。

> **上手行动：** 今晚做 L1——15 分钟完成一次模拟风格锁定（3 张参考图 + 3-5 个主色 + 一句话风格描述），发给朋友做"看得出来是什么游戏吗"测试。下一次 Game Jam 开始前，把这页风格指南打印出来钉在第二屏幕——它就是你 48 小时的 eslint 规则。

### 1.8 扩展阅读

1. **《The Art of Game Design》第 7 章（Jesse Schell）** ——关于"体验由什么构成"的经典论述，帮你理解为什么视觉一致性格外重要。
2. **GDC Talk: "Juice It or Lose It"（Martin Jonasson & Petri Purho）** ——15 分钟内展示用简单特效让游戏"看起来更好"的所有关键技巧。
3. **Ludum Dare 赛后复盘区（ldjam.com）** ——海量 Game Jam 参赛者的复盘文章，搜索 "art" 或 "visual" 找视觉相关复盘。
4. **制作01《你的工具：Aseprite 与工具逻辑》** & **制作05《动画与特效》** ——**为什么推荐：** 本章把像素定为 GameJam 首选，前提是你已经走完制作01 的 Aseprite 工具逻辑和制作05 的动画基础。如果还没，先回这两章——48 小时里你没有"边学工具边 jam"的预算。
5. **风格04《视觉风格文档》** ——**为什么推荐：** 本章 1.2 第 1 小时的"风格指南 1 页"就是风格04 三档模板里最快的那档。赛后想把风格指南升级成完整项目文档，回风格04。
