# 制作03 环境与 Tile：可信世界的三层搭建

### 3.0 这一章解决什么问题

制作02 收尾时，你的贯穿角色跑完了 10 步管线的最后三步，存成 `character-final.png` 拖进了 Godot。可它站在一个空场景里——脚下没有地、头顶没有天、远处没有山。一个角色站在一片平坦的灰色背景前，还没有"世界"。环境与场景（Environment & Scene）是你游戏世界的"身体"——角色需要这个身体来承载，玩家需要这个身体来"相信"。

独立游戏的场景设计有两重矛盾：

1. **你一个人要做完所有场景——但 AAA 游戏的一个场景有一个 5-20 人的"环境美术"团队。** 你的场景制作方式不能是"手工做每一个房子的每一块砖"——你需要系统化的模块化思维。这和你独立写后端不能手写每一个 SQL 查询一样——你要的是 ORM、是脚手架、是可复用的组件。
2. **你的场景需要承载游戏性信息——但你也需要它"美"。** 你的场景不能只是一条"功能性走廊"（玩家从这里走到那里，旁边的墙只是"墙"）——场景中的每一个视觉元素都在说"这里是什么地方？之前发生了什么？之后可能发生什么？"

**本章核心承诺：** 你将学会场景的三层叙事法——前景说"谁的视角"，中景说"正在发生什么"，远景说"暗示什么"。你将把书 A 的 tileset 模块化三规则（相邻匹配 / 边界处理 / 变体防重复）和书 B 的 Tile 工程化内容（Tile 概念与优势、2 的幂尺寸、无缝纹理 do/don't、9-Slice 与 3-Tile 转换、Tile 库、场景组装）统一成一套模块化构建体系——不是两套各讲一遍，是一个 tile 小节里从概念到过渡方法一次讲完。你将掌握三光源设计（Key/Fill/Ambient）在你的引擎中怎么做、参数怎么设。你将学会关卡视觉节奏的调控——不是把所有关卡做成"亮度一样"的，是用对比度/细节密度/色温三个参数创造视觉呼吸。最后你还会用调色板替换实现换季换时——同一套 tile 零额外绘制出白天/夜晚、春夏秋冬。

> **贯穿式项目衔接：** 练手06 你已经接触过 Tile 的数学本质——`uint8_t tile[16][16]` 的周期性拷贝、无缝 = 周期性边界条件、CSS `background-repeat` 类比。本章把那套"单 tile 的无缝制作"扩展到"多 tile 的模块化场景构建"——从一块砖到一座城。制作02 的 `character-final.png` 将在本章 L1 搭出的第一个 tile 场景里第一次"有地方站"。

### 3.1 场景的三层叙事——前景 / 中景 / 远景

场景不是一张"风景明信片"——它是一个**叙事结构**，有三个层次，每层承载不同的叙事功能。

![场景三层叙事](images/制作/制作03-fig01-三层叙事.svg)

*图 制作03.1：场景三层叙事（Three-Layer Scene Narrative）——一张图里藏着三个故事。前景把玩家"推入"场景。中景承载游戏性。远景暗示世界边界和情绪基调。*

#### 前景（Foreground）——"谁的眼睛在看？"

前景是离"玩家眼睛"最近的视觉层。它通常包含：近处的物体（树、草丛、建筑柱子、岩石的边缘）、画面的自然画框元素（从一个窗框看出去、从树丛间看过去）、以及任何遮挡中景/远景的元素。

**前景的叙事功能：** 前景告诉你**"你在这个场景中的位置"**。你不只是在"看"这个场景——你在场景中"有位置"。一棵近处的树——你在树下。一个近处的墙壁——你站在墙后面窥视。这种"有位置"的感觉让玩家从"局外的观察者"变成"场景中的存在"。

**前景的视觉参数：** 可高度模糊/暗/简化（角色是制造空间感不是传递信息，画复杂细节是浪费）；颜色偏暗创造"景深框"（暗前景 + 亮中景 = 你被"推"进场景）；不应遮挡核心游戏性信息——开关/门/NPC 被前景树遮住 = 关卡功能失效。

> **程序员类比：** 前景 ≈ 你的 IDE 的窗口框架。你不是在"看"你的代码——你是在一个窗口中"看"你的代码。前景给了视觉参照——让你的大脑知道"你的视点在哪里"。没有前景的场景 = 没有 IDE 窗口框架的代码 = 代码漂浮在空气中，你没有"位置感"。

#### 中景（Midground）——"正在发生什么？"

中景是场景的核心——它承载游戏性。角色在这里走、跳、攻击。门在这里被打开。NPC 在这里提供对话。敌人在这里出现。宝箱在这里等待被打开。

**中景的视觉参数：** 最高对比度（中景对比度 > 前景和远景，角色需在背景前清晰可辨——中景高对比装饰太多会"吞"掉角色）；最高细节密度（玩家注视中景最久，细节预算集中在此）；主色板最强应用在游戏性对象上（门/开关/宝箱需更高饱和度或明度对比——视觉上说"点这里"）。

**关键设计规则：** 中景元素传递**两种信息**——"美感信息"和"游戏性信息"。如果"看起来很美但让我搞不清它是不是墙"——美感踩了游戏性。**游戏性信息永远优先于美感信息**——一扇门可以美学朴素，但必须显而易见可以打开。

#### 远景（Background）——"暗示什么？"

远景是场景中离玩家最远的视觉层。它包含：天空、远山、远处的城市天际线、远处的森林、远处的海洋——都是用极其简化的方式表现。

**远景的叙事功能：** 远景告诉你**"这个世界有多大多深"**——"那个远处的城堡可能就是你最终要去的地方"——"那片暗色的森林可能未来是另一个关卡"。远景承载的是**未来的承诺**——让玩家的想象力填充"隐去的空间"。

**远景的视觉参数：** 最低对比度（受大气透视影响，对比度降低、颜色向蓝灰色偏移）；最低细节密度（远山 3 个色块就够，加细节是浪费——加了玩家看不到，或看到了反而破坏距离感）；雾是你最好的朋友（引擎启用景深雾，雾色匹配主色，远景淡入雾中创造"远处还有更多"的景深错觉）。

### 3.2 Tile 概念与模块化构建——一块砖到一座城

这是本章的合并核心。书 A 的"Tileset 模块化三规则"和书 B 的"Tile 概念 / 尺寸 / 无缝 / 9-Slice / 3-Tile / Tile 库 / 场景组装"讲的是同一件事的不同侧面——A 讲的是**模块之间怎么协作**，B 讲的是**模块本身怎么造、怎么拼**。本章把它们统一成一个 tile 小节，从概念到过渡方法一次讲完，不重复讲两遍。

#### 3.2.1 Tile 是什么——可复用的图形组件

**Tile（瓦片）** 是可重复使用的小块像素画，通过拼接可以快速构建大型游戏场景。从《超级马里奥》的地面砖块到《星露谷物语》的泥土方块，核心逻辑都一样：你画一个 N×N 像素的方块，引擎把它在 X 和 Y 方向上无限复制。

练手06 已经给你了 Tile 的数学本质——`uint8_t tile[16][16]` 的周期性拷贝。这里换一个工程视角：

> **程序员类比：Tile = 可复用的图形组件，像 React 组件 / 函数——画一次，实例化 N 次，内存 O(1) 表达 O(N)。** 你不是在场景画布里手画 10000 块草地——你画 1 个草地 tile，引擎在 10000 个网格位置引用它。改这个 tile，10000 个位置同步更新。这和你在 React 里写一个 `<Button>` 组件然后在 100 个页面复用一个道理——组件定义一次，实例化 N 次，改组件所有实例同步变。

Tile 的三大优势（书 B 总结，和上面的组件类比一一对应）：
1. **节省工作量**：画 5 个 tile 就可以搭建一整个关卡（= 写 5 个组件拼出一个页面）
2. **节省内存**：重复使用 tile 意味着文件极小（= 组件复用，不重复打包）
3. **快速迭代**：修改一个 tile，整个场景中所有使用它的位置都会更新（= 改组件源码，所有引用处热更新）

#### 3.2.2 Tile 尺寸——2 的幂是工程约定

最常见的 tile 尺寸是 16×16、32×32 和 8×8。选择时注意：

- **幂次方原则**：tile 尺寸通常是 2 的幂次方（8、16、32、64），这是计算机最擅长处理的大小——位运算对齐、纹理上传 GPU 按 2 的幂分块都依赖这个约定
- **面积关系**：32×32 的 tile 面积是 16×16 的 4 倍，但绘制工作量远不止 4 倍（细节密度非线性增长）
- **风格匹配**：小 tile（16×16）适合精细的场景细节，大 tile（32×32）适合粗犷风格

> **程序员类比：2 的幂尺寸 = 内存对齐约定。** 就像结构体按 4/8 字节对齐让 CPU 访问更快，tile 按 2 的幂让引擎的网格划分、纹理打包、位掩码自动 tile 计算都走快路径。非 2 的幂尺寸不是不能 work——是会让某些引擎优化失效。

**和你的角色尺寸锁定。** 制作02 已经点过这条：同一游戏所有 tile 必须同尺寸，且和角色尺寸匹配。你的角色 32×32，tile 就锁 32×32——混用 16×16 和 32×32 = 碰撞体积和场景网格全乱 = bug 温床。

#### 3.2.3 模块化三规则——相邻匹配 / 边界处理 / 变体防重复

这是书 A 的 tileset 思维核心。tile 不是孤立的——它要和相邻的 tile 协作。

**规则一：相邻匹配（Adjacency Matching）。** 一个 tile 的右边挨着另一个 tile——你的边界线必须和右边 tile 的左边界对齐。画 tile 时要考虑四个边的视觉信息：上/下/左/右各有什么？如果 tile A 的右边和 tile B 的左边不匹配——A 放在 B 旁边时玩家会看到一条"缝"。

**规则二：边界处理（Boundary / Edge Handling）。** 两个不同地形类型（草地和沙地、水面和陆地）的边界需要"过渡 tile"。三种方式：**直线硬切**（仅适合几何化风格——平台跳跃的"方块天地"：草地 → 一像素黑线 → 沙地）；**渐变过渡**（dithering 或 alpha 混合创造 4-8 像素宽过渡区）；**混合过渡**（边界放独立物体如石头/灌木遮挡边界线）。

> **程序员类比：边界处理 = 不同类型系统之间的适配器（adapter）。** 草地 tile 和沙地 tile 是两个"类型"，直接相邻 = 类型不匹配的编译错误。过渡 tile 就是 adapter——左半实现草地接口、右半实现沙地接口，让两个类型在边界处合法对接。

**规则三：变体防重复（Variation to Prevent Repetition）。** 只有 1 个草地 tile 铺满 100×100 场景 = 极其重复的"纹理壁纸"。你需要 2-3 个变体 tile（花朵位置不同、草疏密不同、砖块纹理微移），铺草地时随机选变体 1/2/3 制造"不重复的随机自然感"。**变体数量法则：** 3 个 = 基本防重复；5 个 = 大部分场景"自然感"达标；8+ 个 = 肉眼极难察觉重复——但意味着画 8 个 tile 而不是 1 个。练手06 L2 的 16×16 石头 tile 就是按变体思路做的。

#### 3.2.4 无缝纹理 do / don't——练手06 的延续

**无缝纹理（Seamless Texture）** 是指 tile 的左右/上下边缘可以完美对接，看不出重复的边界。练手06 已经给了你完整的"无缝四步法"（种子 → 边界 → 检验 → 变化）和数学本质（周期性边界条件 / torus 拓扑）。这里只补书 B 的 do/don't 检查表，作为你做完四步法后的验收清单：

![无缝纹理问题](images/制作/制作03-fig05-无缝纹理问题.svg)

*图 制作03.5：无缝纹理的常见失败（左/中）vs 成功（右）。左：边界像素不连续 → 可见接缝线。中：tile 内部有方向性渐变 → 重复后形成规则条纹。右：均匀分布、无焦点、无方向渐变 → 真正无缝。*

| 问题 | 描述 | 解决方法 |
|------|------|---------|
| **突出元素** | 单个石块太明显，重复后成"地标" | 均匀分布，避免焦点 |
| **内部渐变** | tile 内部有左→右渐变 | 重复后会形成规则条纹；让明度中心回到几何中心 |
| **高对比度** | 明暗差异太大 | 降低对比度，使 tile 更柔和 |
| **元素集中** | 所有细节挤在一角 | 均匀分布细节 |
| **边缘强调** | tile 边缘有太多像素 | 注意边缘对齐，避免"接缝" |

**环境 tile 的对比度应该低于角色精灵**——这样背景不会抢夺玩家对角色的注意力。这和 3.1 中景规则呼应：中景对比度 > 前景和远景，而角色对比度又要 > 中景背景——视觉层级是 角色 > 中景背景 > 前景/远景。

> **程序员类比：无缝 = 周期边界条件，像 torus 拓扑——铺贴无接缝。** 你的 tile 在 X/Y 方向无限重复，数学上等价于把 tile 的四条边粘成一个环面（torus）。任何在 tile 边界上不连续的像素，都会在这个环面上变成一条肉眼可见的"裂缝"。练手06 讲过的 Aseprite Tile Mode / Krita W 键就是你的"周期边界检查器"——边画边看到 2×2 或 4×4 重复效果。

#### 3.2.5 9-Slice 与 3-Tile——两种地形过渡方法

这是书 B 的核心贡献。当地形从 A 变到 B（草地→沙地），你需要在边界放过渡 tile。有两种方法，工程量 vs 效果的取舍不同。

![九宫格与三段对比](images/制作/制作03-fig04-九宫格与三段对比.svg)

*图 制作03.4：9-Slice（左）vs 3-Tile（右）两种地形过渡方法。9-Slice 切九宫格，四角固定/四边拉伸/中心填充，一个 tile 适配任意尺寸；3-Tile 只切左/中/右三段，适合 1D 重复（围墙/平台），像 CSS 的 repeat-x。*

**9-Slice 过渡。** 最经典的方法。把过渡区域想象成一个 3×3 的九宫格——四个角是固定 tile（保留光源方向一致性），四条边是拉伸 tile（处理草地→沙地的线性过渡），中心是填充 tile。实际制作时用 8-12 个过渡 tile 处理两种地形之间的所有边界情况（内角/外角/直线/端点）。

**优点：** 可以创建任意形状的地形区域；保留光源方向的一致性；视觉效果最自然。
**缺点：** 需要制作 8-12 个过渡 tile；拼接时要注意光源方向。

> **程序员类比：9-Slice = CSS `border-image`——九宫格切片，四角固定、四边拉伸、中心填充，一个 tile 适配任意尺寸。** 你在 CSS 里写 `border-image: url(frame.png) 30 fill;` 时，浏览器把那张图切成九块，四个角原样放、四条边按元素尺寸拉伸、中心填满剩余空间。9-Slice tile 是同一个算法在游戏 tile 里的实现——一张过渡 tile 适配任意大小的地形块。Unity 的 9-Slice sprites、Godot 的 NinePatchRect 都是这个机制。

**3-Tile 高效方法。** 对于追求效率的独立开发者，可以用 2-3 个 tile 模拟过渡效果：

1. 创建一个对角线切割的过渡 tile（左半草地、右半沙地）
2. 复制并水平翻转得到第二个
3. 再创建一个逆方向的角落 tile

**优点：** 极大地减少了工作量——3 个 tile 替代 9-Slice 的 8-12 个。在大多数人眼中效果已经足够好。
**缺点：** 牺牲了精确的光源方向——翻转 tile 会让光源方向反转（主光在右上，翻转后变左上），有经验的玩家会察觉。

> **程序员类比：3-Tile = 简化版 9-Slice，只切左/中/右三段——适合 1D 重复（围墙/平台），像 CSS 的 `repeat-x`。** 当你的过渡只是一维的（一条围墙的左端 / 中间重复 / 右端），不需要九宫格那么复杂——三段就够。这就像你给一个按钮只切左圆角 / 中间拉伸 / 右圆角三段，不需要九宫格的四个角都独立。3-Tile 是 9-Slice 在 1D 场景下的降级版——工作量减 2/3，适用场景也收窄到 1D。

**怎么选？** 需要任意形状的 2D 地形区域（俯视角的草地/沙地/水池任意形状）→ 9-Slice。只需要 1D 重复的边界（横版平台跳跃的围墙、平台边缘）→ 3-Tile。独立开发者预算紧 → 先 3-Tile 跑起来，项目后期有预算再升级到 9-Slice。

#### 3.2.6 Tile 库与场景组装

**调色板准备。** 开始制作 tile 前，先规划整个场景的调色板——这是练手05 有限调色板在场景层的落地：草地绿 4-5 色、沙地棕 3-4 色、水蓝 3-4 色、石头灰 3-4 色、点缀色（花/装饰物）。

**Tile 库（Tile Set）。** 在画布上保留每个 tile 的"原件"组成 tile 库。Aseprite 里可存为独立 `.ase` 文件，需要时复制粘贴到场景画布。更工业化的做法是用 Tiled 或 LDtk 管理tileset——把 tile 库作为 sprite sheet 引用，场景文件只存 tile 索引（= 数组下标），文件极小。

**场景组装四步：** 1) 用基础 tile 填充整个区域打底 → 2) 边界位置用过渡 tile（9-Slice 或 3-Tile）→ 3) 零星添加变体 tile 打破重复感 → 4) 添加角色和互动元素（你的 `character-final.png` 在这一步进场景）。

> **程序员类比：场景组装 = 用 tile 索引拼装一个 2D 数组地图。** 场景文件本质是 `uint8_t map[H][W]`，每个格子存一个 tile 索引（= 指向 tile 库的下标）。Tiled/LDtk 是这个数组的可视化编辑器——你在网格上点哪个 tile，它就在对应格子里写哪个索引。渲染时引擎读 `map[y][x]` → 查 tile 库 → 画对应 tile——O(1) 查表，O(N) 画面。

### 3.3 光源设计——Key / Fill / Ambient 三步

练手04 讲过三点光照的明度骨架，这里展开为**所有视觉风格的通用光源设计框架**，并给出引擎参数。

![三光源布局](images/制作/制作03-fig02-三光源布局.svg)

*图 制作03.2：三光源场景布局（Three-Point Light Scene Layout）——主光定义形体（右上 30°，暖色），补光照亮暗面（左下，冷色），环境光消除死黑（全局均匀，极淡冷色）。三光的独立性 = 场景视觉的可调节性。*

**主光源（Key Light）——"光和影的方向"。** 主光定义你的世界的光源方向，决定角色/树/建筑投下的影子方向和长度。典型参数：角度右上 30-45°（人类最习惯的光源方向——"光源在下方"会触发本能的"不对劲"感，除非你做恐怖游戏）；颜色暖色 `#FFF2E0`；强度 1.0；必须投射阴影。

**补光（Fill Light）——"照亮黑暗的地方"。** 补光从主光对面方向照射，模拟环境反射光，照亮主光产生的暗面。典型参数：角度主光对面（左下）；颜色冷色 `#D4E6F1`；强度 0.3-0.5（明显比主光弱——不让补光"消掉"主光的阴影效果）；不投射阴影。

**环境光（Ambient Light）——"让最暗的地方也不是纯黑的"。** 环境光是全局均匀的"底光"——不来自任何特定方向，来自所有方向同时。作用是确保场景中没有任何区域 RGB = 0,0,0。典型参数：颜色极淡冷色 `#E8F0FF`；强度 0.1-0.2；无方向。

**在引擎中怎么做：**
- Unity: Key = Directional Light（Rotation 30,45,0，暖白，强度 1.0，Soft Shadows）；Fill = 第二个 Directional Light（强度 0.3）；Ambient = Lighting → Environment → Ambient Color
- Godot: Key = DirectionalLight3D（设置旋转，启用 Shadows）；Ambient = WorldEnvironment → Environment → Ambient Light
- 2D 游戏（非 3D 引擎）：如果你的 2D 引擎没有"光源"系统，你需要在画场景时**手绘光照和阴影**——模拟一个右上方向的主光源。练手04 的明度骨架就是手绘光照的底层训练

> **关键洞察：三光源的独立性 = 场景视觉的可调节性。** 如果你后面的关卡需要"更暗更紧张的氛围"——你不需要重画场景——你只需降低环境光强度从 0.15 到 0.08、增加主光的暖色饱和度让光照对比更强。你在引擎灯光面板做 2 分钟的调整——完成的视觉变化是通过"重画场景"无法定义的超级效率。这就像你调一个 CSS 主题变量——改一处 `--key-light-intensity`，整个场景的 mood 联动变化。

### 3.4 关卡视觉节奏——对比度 / 细节密度 / 色温的三元调控

一个关卡不是一个"亮度带"——是多个视觉段落之间的**呼吸节奏**。AAA 游戏通过复杂的后处理体积和脚本化光照变化来创造视觉呼吸。独立游戏的关卡通常是一连串没有脚本化光照变化的房间/区域——所以你需要用**静态的手绘/建模的视觉参数**来创造呼吸感——不需要脚本。

![关卡视觉节奏](images/制作/制作03-fig03-关卡视觉节奏.svg)

*图 制作03.3：关卡视觉节奏调控（Level Visual Rhythm）——三个区段（安全/紧张/过渡）通过三元参数（对比度/细节密度/色温）的调制创造视觉呼吸。这跟音乐的和弦进行是同一个逻辑——节奏不是速度，是变化模式。*

**三元参数的节奏结构：**

- **安全区（Safe Zone）——"休息点"。** 玩家存档、升级装备、和 NPC 对话。视觉语言："放松、温暖、清晰"。对比度低（眼睛放松）、细节密度中低（不分心）、色温暖色（触发"安全/舒适"心理信号——橙灯、暖木色、金色日落）
- **紧张区（Tension Zone）——"高风险"。** 战斗、解谜、危险导航。视觉语言："危险、警觉、快速"。对比度高（大脑进警觉模式）、细节密度高集中在威胁区（敌人/陷阱高细节，背景中低）、色温冷色（深蓝/冷灰/暗紫触发"警惕/不安"——恐怖游戏也可能用暖色制造不适）
- **过渡区（Transition Zone）——"预备"。** 两个区之间不要硬切——从暖色村庄直接进入冰冷墓穴会让大脑在 0.2 秒内不适。过渡区的视觉语言："中间的"——中等对比度、中性色或渐变色温、细节密度逐渐过渡。"走廊"是天然过渡区

**节奏模式：** `安全区（暖·低对比·低细节）→ 过渡区（中·中对比·中细节）→ 紧张区（冷·高对比·高细节）→ 过渡区 → 安全区 → ...`

> **程序员类比：这节奏和音乐中的"和弦进行"是同一逻辑——变化的不是亮度，是三个参数之间的关系和变化的模式。** 玩家不知道自己感受到的是"对比度变化"——他们只知道"哦，这里是安全的"还是"哦，这里可能有危险"。就像你听音乐不会意识到"这是 I-vi-IV-V 进行"，你只感觉到"副歌来了"。和弦进行不创造音符，创造的是**期待和释放**——三元参数节奏不创造像素，创造的是**紧张和放松**。

**Boss 战前 3-5 秒的"视觉紧张爬坡"：** 在 Boss 房间之前的走廊——Boss 还没出现——但视觉参数已经在"预告"Boss 的出现。这 3-5 秒的"爬坡"让 Boss 的出场不是从零→紧张——而是从"正在变得紧张"→紧张。视觉上表现为：逐增的对比度（走廊越来越暗但光照越来越锐利）、逐增的细节密度（墙上开始出现尖锐几何形）、逐冷的色温（从暖色野外→中性走廊→冷色 Boss 入口）。

### 3.5 调色板替换换季换时——零额外绘制

同一个场景同一个 tileset，通过全局调色板调整就能呈现完全不同的氛围——白天变夜晚不需要重画，春天变秋天不需要重画。

> **程序员类比：换季调色板 = 换主题 CSS。** 同一套 tile 像素数据不变，只换 palette index 的 RGB 值——白天版本把 `PALETTE[3]` 映射到 `#E74C3C`（亮红），夜晚版本把同一个 `PALETTE[3]` 映射到 `#7A1F1F`（暗红）。像素数据不变，只换色板查找表，白天/夜晚零额外绘制。这是练手05 调色板替换在场景层的应用——和制作02 的昼夜调色板替换是同一个机制，一个换角色身份色，一个换场景氛围色。

制作02 的 2.6 已经在角色层讲过昼夜调色板替换的 Aseprite 操作（魔棒选相似颜色区域、全局替换、调容差）。场景层完全一样——只是替换范围更大（整个 tileset 而非单个角色）。配合 3.4 的关卡视觉节奏，你可以在同一个关卡里实现"时间流逝"——清晨暖色 → 正午高亮 → 黄昏橙红 → 夜晚冷暗——四套色板一套 tile，昼夜循环零额外绘制。季节调色板同理：春（鲜绿 `#7DCE82` + 粉花 `#F5B7B1`）→ 夏（深绿 `#27AE60` + 蓝天 `#85C1E9`）→ 秋（橙黄 `#E67E22` + 红棕 `#A04000`）→ 冬（白灰 `#ECF0F1` + 冷蓝阴影 `#AED6F1`）。同一套草地 tile，四套色板出四季——这是独立游戏"小预算大世界"的核心杠杆。

### 3.6 常见踩坑

**踩坑一："我的场景只是一条功能走廊"。** 场景只有"玩家走过的路"和"挡住玩家的墙"，没有前景和远景。解决方案：给每个场景加至少一个"叙事物体"——一个破旧木牌、一棵被砍断的树桩、一个熄灭火堆、一本落在地上的书。1-3 个叙事物体就能从"功能走廊"升级为"世界中的一段"。

**踩坑二："场景太亮或太暗——我看不到角色了"。** 用引擎默认光照——一切被均匀照亮，角色和背景在同一亮度层。解决方案：三光源设计（3.3）——主光让角色投下影子"升起"，补光让暗面不完全黑，环境光确保无死黑。

**踩坑三："我的每个房间/区域看起来都一样"。** 关卡所有区域的对比度/色温/细节密度都一样——视觉上"没有起伏"。解决方案：用三元参数（3.4）为每个区域设计不同视觉模式。

**踩坑四："远景细节画太多——玩家永远看不到"。** 远景中的远山画了 20 个小屋顶和窗户——1× 分辨率下从来看不到。远景最高细节标准是"一个色块 + 一个重要地标（如远处发光的塔）"——余下用雾和简化"隐去"。

**踩坑五："9-Slice 翻转 tile 光源穿帮"。** 用 3-Tile 方法翻转 tile 省工作量，但翻转后主光从右上变左上——和场景其他 tile 的光源方向不一致，有经验的玩家会察觉"这块地的光不对劲"。解决方案：要么补画一个非翻转的版本（多 1 个 tile 工作量），要么在翻转 tile 上手动修光影方向。这是 3-Tile 牺牲精度换工作量的代价——9-Slice 不存在这个问题。

**踩坑六："tile 内部渐变 → 重复后变条纹"。** 你在画草地 tile 时潜意识从左到右做了个"左暗右亮"的渐变——单看这块 tile 没问题，4×4 重复后相邻 tile 的"左暗"紧挨"右亮"，形成规律的明暗棋盘条纹。解决方案：检查 tile 四边平均明度是否一致，让明度中心回到几何中心。**记住：无缝是"每条边在任何位置亮度一致"，不是"整个 tile 亮度均匀"。**（练手06 错误 3 有完整诊断。）

### 3.7 练习

本章练习逐级递进，每级有明确"合格标准"。三个练习建议分三天。

#### L1 · 第一个 tile 场景：3 地形 + 3-Tile 过渡（约 90-180 分钟）

**目标**：用 Aseprite 搭建一个包含 3 种地形（草地/沙地/水）的 8×6 小场景，用 3-Tile 方法处理过渡，最后把制作02 的 `character-final.png` 放进去验证"有地方站"。**你需要**：Aseprite、制作02 的 `character-final.png`（没有就用任何 32×32 角色 sprite）。

**步骤**：
1. **调色板准备**：新建 32×32 画布，准备调色板——草地绿 4 色、沙地棕 3 色、水蓝 3 色、点缀色 2 色（花/石头）。
2. **基础 tile**：每种地形画 1 个基础 tile（32×32）。草地用随机抖动暗示草叶，沙地用均匀颗粒，水面用平行线暗示波纹（练手06 抖动全谱）。
3. **变体 tile**：每种地形再画 2 个变体（挪细节位置 / 加小装饰），共 9 个基础 + 变体 tile。
4. **无缝检验**：每个 tile 复制成 4×4 排列，眯眼看有无接缝或条纹（练手06 四步法 Step 3）。
5. **3-Tile 过渡**：按 3.2.5 的 3-Tile 方法做草地→沙地、沙地→水的过渡 tile（对角线切割 + 水平翻转 + 逆方向角落，每个过渡 3 个 tile）。
6. **场景组装**：新建 8×6 格 = 256×192 画布，按 3.2.6 四步组装——基础填充 → 过渡 tile 放边界 → 变体打破重复 → 留一个格子放角色。
7. **角色入場**：把 `character-final.png` 放进场景，缩放对比——角色清晰可辨？色彩冲突？角色是否"融入"背景消失？
8. 导出场景 PNG，存为 `scene-v01-tile.aseprite` + `scene-v01-tile.png`。

**合格标准**：
- [ ] 3 种地形基础 tile 各 1 个 + 各 2 变体，共 9 个 tile，全部 4×4 无缝检验通过。
- [ ] 2 组 3-Tile 过渡（草→沙、沙→水）各 3 个 tile，共 6 个过渡 tile。
- [ ] 8×6 场景组装完成，无明显接缝、无规律条纹、变体打破了重复感。
- [ ] 角色放入后清晰可辨，色彩不冲突——角色对比度 > 中景背景。
- [ ] 文件存为 `scene-v01-tile.aseprite` + `scene-v01-tile.png`。

#### L2 · 9-Slice vs 3-Tile 对比实战（约 60-120 分钟）

**目标**：用同一种地形过渡（草地→沙地）分别做 9-Slice 和 3-Tile 两套过渡 tile，并排放在同一个场景里对比——亲眼看出 9-Slice 的光源一致性优势和 3-Tile 的工作量优势。

**步骤**：
1. 复用 L1 的草地和沙地基础 tile。
2. **9-Slice 版**：按 3.2.5 做 8-12 个过渡 tile（内角/外角/直线/端点），注意保持主光右上 30° 方向一致。
3. **3-Tile 版**：按 3.2.5 做 3 个过渡 tile（对角线 + 翻转 + 逆角落），注意翻转 tile 的光源穿帮问题。
4. 在 16×6 画布的左半用 9-Slice、右半用 3-Tile 各搭一个 8×6 的草地→沙地过渡场景。
5. 眯眼对比：9-Slice 的过渡是否更自然？3-Tile 的翻转 tile 是否能看出光源穿帮？

**合格标准**：
- [ ] 9-Slice 8-12 个过渡 tile 制作完成，光源方向一致。
- [ ] 3-Tile 3 个过渡 tile 制作完成。
- [ ] 两套并排对比图导出，能指出 9-Slice 优势（光源一致/形状任意）和 3-Tile 优势（工作量 1/3）。

#### L3 · 调色板替换：同一场景白天 / 夜晚 / 四季（约 30-60 分钟）

**目标**：用 L1 的 `scene-v01-tile.aseprite`，通过调色板替换生成白天 / 黄昏 / 夜晚三版 + 春 / 秋 / 冬三版，零额外绘制。

**步骤**：
1. 复制 `scene-v01-tile.aseprite` 为 `scene-v02-day.aseprite`（白天原版）、`scene-v02-dusk.aseprite`、`scene-v02-night.aseprite`。
2. **黄昏版**：Aseprite 魔棒选草地绿，全局替换为橙黄色（`#7DCE82` → `#E5C46B`）；天空/水蓝替换为橙红（`#85C1E9` → `#E67E22`）。调容差控制选择范围。
3. **夜晚版**：草地绿替换为暗蓝绿（`#7DCE82` → `#3A5A40`）；沙地棕替换为冷灰棕；水蓝替换为深靛蓝；整体降低明度 30-40%。
4. **季节版**（可选）：春（原版鲜绿 + 粉花点缀）、秋（草绿→橙黄 `#E5C46B` + 红棕 `#A04000`）、冬（草绿→白灰 `#ECF0F1` + 冷蓝阴影 `#AED6F1`）。
5. 6 个版本并排导出 PNG。

**合格标准**：
- [ ] 白天/黄昏/夜晚三版生成，同一场景三种氛围，零额外像素绘制（只换色板）。
- [ ] 角色在夜晚版仍清晰可辨——没有因为场景变暗而角色消失（角色的明度三步保证了跨背景可见性，制作02 2.4）。
- [ ] 季节三版（春/秋/冬）生成，同一 tileset 出四季。
- [ ] 6 版并排对比图导出。

### 3.8 检查点

1. **我的场景有三层叙事吗？** 前景给位置感、中景承载游戏性、远景给世界感——三层关掉各一层都能发现问题。
2. **我的 tile 全部 4×4 无缝检验通过吗？** 无接缝、无条纹、无地标式焦点（3.2.4 + 练手06）。tile 尺寸和角色锁定一致吗（角色 32×32，tile 锁 32×32）？
3. **我用了 9-Slice 还是 3-Tile？选择理由清楚吗？** 2D 任意形状 → 9-Slice；1D 重复 → 3-Tile；预算紧先 3-Tile。3-Tile 翻转 tile 的光源穿帮我处理了吗？
4. **我的三光源在引擎里设好了吗？** Key 右上 30° 暖色 1.0 投阴影、Fill 对面冷色 0.3-0.5、Ambient 全局极淡冷色 0.1-0.2。
5. **我的关卡有三元参数的节奏起伏吗？** 安全/紧张/过渡三区在对比度/细节密度/色温上拉开。Boss 战前有 3-5 秒视觉紧张爬坡吗？
6. **我试过调色板替换换季换时了吗？** 同一套 tile 出白天/夜晚/四季，零额外绘制。
7. **我的角色放进场景清晰可辨吗？** 角色对比度 > 中景背景 > 前景/远景——视觉层级清晰。

### 3.9 本章小结

- **场景是三层叙事结构，不是风景明信片。** 前景给位置感（IDE 窗口框架）、中景承载游戏性（游戏性信息 > 美感信息）、远景给世界感和未来承诺（雾和简化隐去细节）。
- **Tile = 可复用图形组件 = React 组件 / 函数——画一次，实例化 N 次，内存 O(1) 表达 O(N)。** 三大优势：节省工作量、节省内存、快速迭代。尺寸按 2 的幂（= 内存对齐约定），且和角色尺寸锁定一致。
- **模块化三规则统一了 A 的 tileset 思维和 B 的 tile 工程。** 相邻匹配、边界处理（adapter 让不同地形合法对接）、变体防重复（3 个基本、5 个达标、8+ 难察）。
- **无缝 = 周期边界条件 = torus 拓扑。** 练手06 给了四步法，本章补了 do/don't 检查表——突出元素/内部渐变/高对比/元素集中/边缘强调是五大失败模式。
- **9-Slice = CSS `border-image`，3-Tile = CSS `repeat-x`。** 9-Slice 切九宫格适配任意尺寸 2D 地形（光源一致但工作量大）；3-Tile 切三段适合 1D 重复边界（工作量 1/3 但翻转 tile 光源穿帮）。
- **三光源 = 场景的光照骨架。** Key 定义方向、Fill 填暗面、Ambient 消死黑。三光的独立性 = 视觉的可调节性——调灯光面板 2 分钟 = 重画无法定义的 mood 变化。
- **关卡视觉节奏 = 和弦进行逻辑。** 对比度/细节密度/色温三元参数起伏——安全区暖低少、紧张区冷高多、过渡区中间渐变。Boss 前 3-5 秒爬坡让出场不是从零而是从"正在变紧张"。
- **调色板替换 = 换主题 CSS。** 同一套 tile 换 palette index 的 RGB 值——白天/夜晚/四季零额外绘制。和制作02 昼夜调色板替换同一个机制。

> **如果只记住一句话：** 场景的"美"不在细节密度——在**信息层级**——玩家的眼睛必须在第一秒看到"往哪儿走/哪有危险/哪儿可以停下"。tile 让你用 O(1) 的组件表达 O(N) 的世界，三光源给你 2 分钟调 mood 的杠杆，三元参数节奏让关卡像和弦进行一样有呼吸。

> **模块化资产清单：** 地面、墙、转角、门、可交互物、叙事小物、前景遮挡、远景地标——8 类 tile 覆盖一个场景骨架。

**上手行动**：今晚按 L1 搭你的第一个 tile 场景——3 地形 + 3-Tile 过渡 + 把 `character-final.png` 放进去。你的角色从制作02 起就在等"有地方站"——这一章给它一块地、一片天、一条地平线。完成后按 L3 换一套夜晚色板——零额外绘制出夜版，你会立刻理解"调色板替换是独立游戏最大的生产力杠杆"是什么意思。

### 3.10 扩展阅读

1. **[Gurney, James. 《Color and Light》](http://jamesgurney.com/)** — **为什么推荐：** 3.3 三光源设计（Key/Fill/Ambient）的核心在 Gurney 的书中有完整的科学和艺术基础展开。
2. **[Solarski, Chris. 《Drawing Basics and Video Game Art》](https://www.chrissolarski.com/)** — 前育碧概念艺术家。**为什么推荐：** 3.1 三层叙事法基于 Solarski 对游戏场景"视觉层级和叙事"的讨论。
3. **[Hodent, Celia. 《The Gamer's Brain》](https://www.gamasutra.com/)** — 认知心理学与 UX。**为什么推荐：** 3.4 关卡视觉节奏的认知心理学验证基于 Hodent 的"认知过载"和"视觉搜索模式"研究。
4. **[Tiled Map Editor](https://www.mapeditor.org/)** & **[LDtk](https://ldtk.io/)** — 2D tile 地图编辑工具。**为什么推荐：** 3.2 tile 库和场景组装的实践工具——Tiled 是事实标准，LDtk 是现代替代品，与 Godot/Unity/Phaser 无缝集成。
5. **[Lospec — Tileset Tutorials](https://lospec.com/)** — **为什么推荐：** "相邻匹配"和"边界处理"规则在 Lospec 社区有大量像素级实现案例。
6. **[Unity Manual — 9-Slice Sprites & Lighting](https://docs.unity3d.com/)** & **[Godot Docs — NinePatchRect & 3D Lighting](https://docs.godotengine.org/)** — **为什么推荐：** 9-Slice（Unity Sprite Editor / Godot NinePatchRect）与三光源在引擎中的技术实现——都是 CSS `border-image` 在游戏引擎里的对应物。
7. **[A Short Hike 的环境美术分析](https://ashorthike.com/)** — **为什么推荐：** "一个人用极简模块化+光照+雾构建完整可信世界"的最佳独立案例。
8. **练手06《质感》** & **制作02《像素角色工作流》** — **为什么推荐：** 练手06 给了 tile 数学本质（周期边界条件）、无缝四步法、抖动全谱——本章在此基础上扩展到多 tile 模块化构建。制作02 的 `character-final.png` 是本章 L1 场景里的角色，其昼夜调色板替换和本章 3.5 换季换时是同一个机制。
