# 制作04 UI 设计：不只是好看，是好用

### 4.0 这一章解决什么问题

制作03 收尾时，你的贯穿角色终于"有地方站"。可玩家还是没法和这个世界交互：还剩多少血？手里是什么道具？那扇门能不能开？这些"游戏内部数据 → 玩家一眼就懂"的翻译，全是 UI 的活。

独立游戏 UI 的三重矛盾：① 你只有一个人，但 UI 是程序+美术+交互的交叉领域；② HUD 在抢玩家对游戏世界的注意力，但你又需要它传递关键信息；③ 约 8% 的玩家有色觉障碍——你的 UI 不能只靠颜色传递信息。

本章按一条递进流程组织：先理解 UI 的三重功能，再建立信息层级概念，然后进入具体设计——从四类型分类到 8 步工作流，最后落到像素 UI 的实战制作。

---

### 4.1 UI 的三重功能——信息 / 操作 / 情绪

UI 是你和玩家之间的对话界面，分三层功能：

**信息（Information）——"告诉你什么"。** 血量、弹药、道具、任务目标。信息必须能在 0.3 秒内被提取。UI的位置要固定。血量——永远在同一个位置。小地图——永远在同一个角落。弹药数——永远在同一个位置。快速反应用图形，精确决策加数字。显示"87/100 HP"在扫视时比一根红色细条更精确。"少即是多"的信息经济学。 一瞬间玩家能处理的信息极其有限，你的 HUD 应该只显示"现在绝对必要"的信息，而不是"所有可能有用的信息"。

**操作（Operation）——"接收玩家的输入"。** 按哪里、点哪个。可交互元素与不可交互元素必须有视觉差异——不能只靠颜色（色盲不友好），需要颜色+形状+明度+边框多重编码。点击/触摸区域必须足够大。反馈必须在 100ms 内出现。

**情绪（Emotion）——"让你感受到什么"。** 你的 UI 的风格、动画、音效——在所有时间都在告诉玩家"你现在在什么样的游戏世界中"。一个纯文本的"HP: 87/100"和一个有脉搏跳动的心脏图标——这两个东西传递的信息完全相同——但前者说"你在操作一个数据"，后者说"你在一个危险的地牢中，你的生命在跳动"。血溅屏幕、暗角、UI材质、Juice 微交互。这些不是装饰——是让玩家"感受到"信息，而不仅是"读取"信息。

> **一句话总结：** 信息让你知道 / 操作让你行动 / 情绪让你相信。

---

### 4.2 信息层级——Primary / Secondary / Tertiary

在画任何 UI 之前，先做一件事：把你 HUD 上的所有信息分成三层。它直接决定玩家扫视时的阅读顺序。

| 层级 | 定义 | 例子 | 位置要求 |
|------|------|------|---------|
| **Primary** | 玩家必须始终能看到的信息 | 血量、BOSS 血条 | 固定、靠近屏幕中央 |
| **Secondary** | 玩家间歇需要参考的信息 | 小地图、弹药数、任务目标 | 角落、边缘 |
| **Tertiary** | 只在特定事件出现的信息 | 伤害数字、获得物品通知 | 出现后消失、不常驻 |

**硬约束：** Primary 层不超过 3 个元素。如果你的 Primary 有 7 个——玩家在同时监控 7 个数据点，工作记忆无法做到。

**自检方法：** 截一张游戏截图，退后两步看 3 秒。你第一眼看到的是 Primary 吗？第二眼是 Secondary 吗？如果不是——调位置和对比度。这个自检比任何 UI 理论都可靠。

---

### 4.3 80/20 法则——HUD 的每一个像素都在抢注意力

眼动追踪研究发现：玩家仅 20% 的视觉注意力花在 HUD 上——另外 80% 在游戏世界中。

![HUD注意力80-20](../images/制作/制作04-fig02-HUD注意力80-20.svg)

三个推论：
- **HUD 元素不是越多越好。** 每个元素都在"偷"玩家对游戏世界的注意力。加了"环境温度显示"但游戏没有温度机制——偷了注意力却给零价值回报。
- **位置决定被注意到的概率。** 屏幕中央区域 = 玩家注视最久，边缘 = 扫视频率低。血条靠中上，小地图放角落，弹药数字在准星附近（不是角落）。
- **重要信息需要多重编码。** 血量低 → 血条变短变红（视觉）+ 心跳音效（音频）+ 手柄震动（触觉）。单一通道 = 单点故障。

---

### 4.4 UI 四类型——从纯屏幕到纯世界

同一个"血量"信息有四种显示方式，按"世界化程度"从低到高排列：

![UI四类型](../images/制作/制作04-fig01-UI四类型.svg)

**Non-diegetic（非叙事 UI）：** 纯屏幕叠加——血条、小地图、技能栏贴在屏幕最上层。最清晰、实现最简单。2D 像素游戏的默认选择。Dead Cells 的 HUD、Celeste 的草莓计数都是这类。

**Diegetic（叙事 UI）：** UI 存在于游戏世界内部——角色能"看到"它。《Dead Space》主角脊背上的血量发光管是教科书案例。沉浸感极高但实现成本大，独立像素游戏谨慎选用。

**Spatial（空间 UI）：** UI 在 3D 空间中悬浮。2D 像素游戏基本用不到——它依赖 3D 透视。

**Meta（元 UI）：** 屏幕效果——受击闪红、低血量暗角、死亡闪白。最强的情绪感染力，实现成本不高（全屏贴图或一个 ColorRect）。独立像素游戏最常见组合：**Non-diegetic（主 HUD）+ Meta（伤害/濒死反馈）**。

### 4.5 8 步 UI 工作顺序

这 8 步不可跳级。前一步没过不碰后一步。

![UI工作流8步](../images/制作/制作04-fig03-UI工作流8步.svg)

**第 1 步：布局层次。** 不要画任何东西。先按 4.2 的三层信息层级把每个 UI 元素归入 Primary / Secondary / Tertiary。Primary 不超过 3 个。

**第 2 步：安全区。** UI 核心元素放在屏幕的 90% 安全区内（边缘各缩进 5%）。手机上额外避开刘海和手势条。

**第 3 步：先建 HUD。** 用纯色矩形 + 占位文本在引擎中搭 HUD 框架（Godot中可先用 ColorRect + Label占位）。先让数据从游戏逻辑流到 HUD（扣血 → 血条变短），美化后面再说。

**第 4 步：测试可读性。** 在不同条件下测试可读性，例如退后两米看、户外背光处看、角色高速移动中尝试读取——任一条件不可读就回第 1 步。

**第 5 步：色盲模拟。** 模拟红色盲和绿色盲进行测试。UI 转灰度后信息还在吗？如果消失——说明你在只靠颜色传递信息。

**第 6 步：添加 Juice。** 让UI变得有生命。例如血条在血量变化时闪一下。经验条填充时微冲过头再回弹。按钮按下缩小 5% 再弹回。这些微交互是"游戏感"的原材料——没有 Juice 的 UI = 操作 Excel。

**第 7 步：集成无障碍。** 至少做 3 件事：字体大小选项、高对比度模式、色盲模式（用形状/图案代替纯色）。像素字体切换大小时保持整数倍缩放。这些工作可以兼容视力较弱的玩家。

**第 8 步：观察真人使用。** 找一个人玩，不说话不解释只观察。哪里犹豫？哪里点错？哪里完全忽略 UI 提示？这一步发现的问题——你靠自己永远发现不了。

### 4.6 像素 UI 资源制作

一个完整的像素 UI 需要哪些 sprite？下面是标准清单：

**HUD 核心资源：**

| 资产 | 包含的 sprite | 像素要点 |
|------|-------------|---------|
| **血条** | frame（边框）+ fill（填充条）+ background（底色） | fill 用高饱和色，bg 用低明度灰色。至少 2 个明度阶 |
| **经验条** | frame + fill | 同血条，颜色区分（蓝/黄/紫） |
| **按钮** | normal + hover + pressed + disabled 四态 | 像素按钮四态只需改 3-5 个像素——top 边框从亮改暗 = pressed |
| **对话框** | 9-Slice 面板（四角固定 + 四边拉伸 + 中心填充） | 复用制作03 的 9-Slice 机制 |
| **图标** | 16×16 或 32×32，和角色尺寸锁定一致 | 换色 = 不同状态（灰=冷却中/亮=可用），零额外绘制 |
| **光标** | 像素箭头或手型指针 | 1-2 色足够 |
| **通知弹窗** | 面板 + 图标 + 关闭按钮 | 入场/退场动画——从透明淡入、向上滑出 |

**Godot 对应节点树示例：**

```
Control (根)
├── HUD (CanvasLayer)
│   ├── HealthBar (HBoxContainer)
│   │   ├── Background (ColorRect)
│   │   ├── Fill (ColorRect — 用 scale.x 控制长度)
│   │   └── Frame (TextureRect — 像素边框 sprite)
│   ├── AmmoCounter (Label — 像素字体)
│   └── MiniMap (TextureRect — 放在角落)
├── DialogBox (Panel — 9-Slice)
│   ├── Background (NinePatchRect)
│   ├── Text (Label)
│   └── CloseButton (TextureButton — 含四态 sprite)
└── Notifications (VBoxContainer)
    └── NotificationItem (HBoxContainer)
```

**关键规则：** UI 色板可以独立于场景色板，但必须在色温上保持一致——场景是暖色调，UI 不能突然变成冷蓝。风格04 的视觉风格文档里有一栏专登 UI 风格决策——把上面的 sprite 清单和色板填进去。

### 4.7 像素 UI 风格

你的 UI 不仅是功能——它也是风格。像素游戏的 UI 必须和画面共用一套视觉语言，风格04 的视觉风格文档里专门有一栏登记 UI 风格决策——这一节给你那栏填什么。 

**核心原则：UI 和游戏画面是同一个"视觉宇宙"。** 像素画面 + 矢量 HUD = 玩家眼睛在两种视觉语言之间反复跳——这种"视觉打架"会在制作09 一致性审计时被抓出来。 

**边框。** 用 1 像素宽的线——和背景有至少 3 个明度等级的差距，否则边框被背景吃掉。边角保留像素特有的锯齿，不做 AA 柔化——这和练手08 的抗锯齿原则一致：UI 边框适合硬边，柔化后反而失去像素识别度。对话框、菜单框、血条外框全部用同一套边框规则——边框宽度和颜色在风格文档里锁死，不因"这个框比较大"就改用 2px 边框。 

**图标。** 图标尺寸和角色/tile 尺寸锁定一致。制作02 锁了 32×32 角色，UI 图标就别突然用 24×24——混尺寸等于视觉打架。图标尺寸的"可读尺度"： 

- 16×16：极简符号——一个心形=血量，一把剑=攻击，一枚金币=金钱。适合信息密度高的面板（背包/商店）。 
- 32×32：可以有更多细节——心形加数字、武器加纹理、技能图标可区分"火球"和"冰箭"。适合 HUD 主界面。 
- 换色区分状态：同一图标换个色板条目 = 不同状态（灰=冷却中，亮=可用，闪=即将过期）。和练手05 的调色板替换是同一机制，零额外绘制。 

**字体。** 必须用像素字体，不能用矢量字体。矢量字体的平滑边缘在像素画面中像"另一个世界的产物"。常见开源像素字体：Press Start 2P、VT323、方正像素 12px。像素字体的可读性依赖整数倍缩放——非整数倍 = 糊。中文像素字体的最小可读尺寸是 14-16px——如果你的 UI 面板需要大量中文，确认像素字体在这个尺寸下可读。

 **面板框架。** 像素 UI 的对话框/菜单框直接复用制作03 的 9-Slice——四角固定、四边拉伸、中心填充，和 tile 的 9-Slice 是同一个算法。菜单背景的"材质暗示"也要统一——奇幻 RPG 用木板/羊皮纸纹理的像素化版本，科幻用金属/蓝光面板的像素化版本。但无论什么材质，它必须是像素画的——不能引入手绘纹理或矢量渐变。 

**面板内信息排布。** 像素 UI 的面板空间非常有限——一个 32×32 的图标区域只有 1024 像素。信息排布遵循：最重要的信息放左上（Z 型扫视的起点），操作按钮放底部或右侧（手指/鼠标最容易到达的位置），装饰元素放四角——不占据信息区的像素。如果一个面板"太挤"，优先删装饰元素而不是缩文字。 

**案例拆解：** 《Stardew Valley》的菜单——木板纹理边框 + 像素图标 + 像素字体，背包/工具栏/对话框和农场画面共用一套像素语言，是像素 UI 一体化的标杆。 《Undertale》的黑底白边像素对话框 + 打字机文字 + 四个大按钮（FIGHT/ACT/ITEM/MERCY）——像素 UI 操作编码的典范：4 个操作各占一个象限，不用读文字看颜色就知道"战斗/行动/道具/慈悲"。 《Dead Cells》的 HUD——暗色高对比像素 UI 配快节奏战斗，血条和技能冷却的信息编码在像素 UI 里做到了"瞟一眼就知道"的速度。 《Celeste》——极简到左上角只有草莓计数和死亡数，像素美学推到极致：UI 几乎消失，融入画面。

### 4.8 色盲无障碍——不纯靠颜色

约 8% 男性有色觉障碍。四条原则：

1. **形状/图标双重编码。** 不要只用"绿=安全/红=危险"——绿色上叠加盾牌图标、红色上叠加骷髅图标。
2. **明度双重编码。** 危险信息标为暗+红，安全信息标为亮+绿——明度差异在色盲条件下仍然可辨。
3. **位置/模式双重编码。** 不可用按钮不仅是灰色——同时去掉边框、去掉投影、文字变细。多重缺陷叠加 = 色盲玩家也能 0.1 秒识别。
4. **音频+触觉多通道。** 血量低——血条变红（视觉）+ 心跳音效（听觉）+ 手柄震动（触觉）。三个通道任一可用，信息就到。

### 4.9 常见踩坑

**踩坑一："别人游戏里有的 UI，我也需要有。"** 每个 UI 元素必须有你的游戏中的存在理由——不是"别的游戏有"。每加一个元素，偷走一部分那仅有的 20% 注意力。

**踩坑二："美化优先于功能"。** 先设计好看框架再填内容 = 返工。内容先行（第 1 步），美化最后（第 6 步）。纯色矩形阶段改位置零成本。

**踩坑三：不同屏幕下 UI 被裁切。** 不要用像素绝对坐标。用百分比定位 + 锚点系统（Godot 的 Anchor / Unity 的 Canvas Scaler）。

**踩坑四：Juice 过量变成噪音。** 任何 UI 元素在安静时不做动画，只在状态变化时做一次。血条永久闪烁 = 玩家 2 分钟后把它当背景噪音——真正"快死了"时反而被无视。

### 4.10 练习

#### L1 · HUD 纯色矩形原型 + 3 秒扫视测试（15 分钟）

截一张游戏画面，用纯色矩形标注 HUD：红色=Primary、蓝色=Secondary、灰色=Tertiary。退后两步看 3 秒——红色先被注意到吗？不是就交换位置。15 分钟纯色矩形验证 = 省掉几小时"画完才发现位置错了"的返工。

#### L2 · 画出你的 UI Asset 清单（30 分钟）

按 4.6 的清单列出你游戏需要的全部 UI sprite：血条几色、按钮几态、对话框尺寸、图标尺寸。画出血条的 3 段 sprite（frame + fill + background）和按钮的 4 态 sprite。全部存进 `assets/ui/` 目录。

#### L3 · 在 Godot 里搭出完整 UI 节点树（1-2 小时）

按 4.6 的节点树示例，在 Godot 里搭出完整 UI 结构。数据流走通——扣血时血条变短、获得弹药时数字增加。Juice 加上——血条变化时闪一下、经验条填充时微冲过头。

### 4.11 本章小结

- **UI 三重功能：信息+操作+情绪。** 信息让你知道、操作让你行动、情绪让你相信。
- **信息层级是第一原则。** Primary/Secondary/Tertiary 三层分配——Primary 不超过 3 个。退后两步 3 秒扫视自检。
- **80/20 法则：HUD 只有 20% 注意力。** 每个元素都要有存在理由。多重编码 = 冗余信号通道。
- **8 步 UI 工作流 = 不可跳级的 pipeline。** 布局 → 安全区 → HUD 先行 → 可读性 → 色盲 → Juice → 无障碍 → 真人观察。
- **像素 UI 制作：** 血条=frame+fill+bg 三段，按钮=四态各改 3-5 个像素，面板=9-Slice 复用制作03。Godot 里 Control→CanvasLayer→HBoxContainer 搭节点树。

> **如果只记住一句话：** UI 的"美"不是第一次设计的目标——"可用"才是。先用纯色矩形验证信息层级，再画像素装饰。
