# 制作08 生产管线：从一个素材到一套资产

### 8.0 为什么需要生产管线

制作07 你把单张资产像素完美地搬上了引擎屏幕。但如果你开始做一个真正的游戏项目，一个尖锐的问题会立刻浮现：**你会画一个宝箱，但你能画 50 个宝箱吗？**

你有一个角色，但它需要待机、走路、攻击三套动画。你改了一版、两版、三版。文件夹里开始出现 `最终版-final-真的最终-v3-副本.png`。你分不清哪个文件是最新的。你花了三个小时画了一个道具，但它和别的道具风格不统一，你得返工。

这就是单个资产能力和**生产系统能力**的差距。很多人会画，但不会"生产"。两者的区别就像"会写一个函数"和"能管理一个代码仓库"——后者需要的不是你更聪明的手，而是一套让你不会搞砸自己的流程。

> **程序员类比：** 这就像你刚开始编程时，一个 `main.py` 文件写到底。后来你学会了模块化、命名规范、Git、CI/CD。不是说后者写出的代码"更正确"，而是后者让你能在凌晨三点找到三个月前写的那行 bug 源于哪个文件。美术生产管线是同一回事——它不是让你画得更好，是让你少踩自己的坑。

### 8.1 一张资产的完整生命周期

在你开始画任何东西之前，先建立"一个资产有六个阶段"的认知。这不是教条，但如果你跳过某个阶段，你几乎一定会在后面付出代价。

![资产生命周期](../images/制作/制作08-fig01-资产生命周期.svg)

*图 8.1：一张像素资产的完整生命周期——参考→草稿→粗胚→精修→导出→引擎验证。*

**阶段一：参考收集。** 至少收集 3-5 张参考图。一张看整体造型，一张看材质细节，一张看色彩方案。参考图用于分析，不直接描摹或商用复制。检查点：你的参考够具体吗？"宝箱"可以是一个铁皮橡木箱，也可以是一个镶宝石的圣物箱——"具体"意味着画的时候不需要再查。

**阶段二：草稿。** 用最粗糙的方式快速出 2-3 个方案（24×24 或 32×32 缩略图，每个 5-10 分钟）。只关心轮廓和明暗大块。检查点：剪影测试——纯黑轮廓能分清是宝箱还是石头吗？这个阶段推翻一个方案只要 5 分钟，在精修阶段推翻要 5 小时。

**阶段三：粗胚。** 定下大型。画出主要体块和固有色，光线只做大致的明暗区分（二分法）。检查点：比例已经定死了吗？色块分区读得懂吗？在这个阶段之后，不应该再做大的比例改动——就像写完代码架构之后不改接口签名。

**阶段四：精修。** 最花时间的阶段。加入细节——纹理、做旧、高光、材质区分、边缘处理、抗锯齿。反复对照参考图。检查点：和参考图对比，细节密度是否匹配？有没有漏画的部分？

**阶段五：导出。** 裁剪到正确边界，确认命名规范，导出 PNG（Nearest Neighbor + 整数倍缩放 + Alpha 通道）。导出阶段请遵循制作07 的全部规范——PNG、Filter、Padding 等设置已在制作07 完整覆盖，本章不再重复。

**阶段六：引擎验证。** 拖进 Godot，放在实际游戏场景中看效果。跑制作07 的 5 件检查（Filter 关、整数缩放、锚点统一、Alpha 无白边、帧序无错乱）。永远在目标环境中做最终判断——Aseprite 里好看 ≠ 引擎里好看。

**时间分配参考：**

| 阶段 | 时间占比 | 说明 |
|------|---------|------|
| 参考收集 | 10% | 前期投入，后期省时间 |
| 草稿 | 15% | 探索方向，成本极低 |
| 粗胚 | 20% | 定型关键，不可跳过 |
| 精修 | 35% | 最花时间的阶段 |
| 导出 | 10% | 规范比速度重要 |
| 引擎验证 | 10% | 最后的防线 |

前三个阶段加起来占近一半时间——很多人把它们压缩到 10%，这就是为什么精修阶段要反复返工。

### 8.2 命名的艺术 + Definition of Done

文件名是同一个道理——当你有一百个资产文件时，`未命名-3.png` 就是一段你不愿意维护的遗留代码。

![命名规范](../images/制作/制作08-fig02-命名规范.svg)

**命名公式：** `类型_对象_动作/状态_序号.扩展名`

| 命名 | 含义 |
|------|------|
| `spr_hero_idle_01.png` | 精灵_英雄_待机_第1帧 |
| `spr_hero_walk_03.png` | 精灵_英雄_走路_第3帧 |
| `tile_grass_01.png` | 图块_草地_01 |
| `ui_btn_start_hover.png` | UI_按钮_开始_悬停态 |
| `item_potion_health_01.png` | 道具_药水_回血_01 |

类型前缀（`spr_` / `tile_` / `ui_` / `item_`）是你的"命名空间"——把不同种类的资产在文件管理器里天然分开。全小写、全下划线、用英文单词——空格需要转义、中文在部分引擎中不稳定、大小写在 Linux/Windows 间歧义。

**从第一个资产开始就用规范命名——这不是"以后再说"的事。**

#### Definition of Done：一张资产什么时候算完成？

不是"我觉得不错"。而是下面所有方框都打勾：

- [ ] 命名按 `类型_对象_动作_序号` 公式完成
- [ ] 导出 PNG，整数倍缩放，Nearest Neighbor
- [ ] Godot 导入后 5 件检查全过（制作07）
- [ ] 放其他资产旁边，风格一致——看不出"这是另一个人画的"
- [ ] Git 已提交（`.ase` 源码 + `.png` 编译产物）
- [ ] 版本文件夹结构完整（`current/` 指向这一版）

全部打勾 = Done。任何一条没勾 = 没完成。这和你的 CI pipeline 里 "all checks passed" 是一个逻辑——不确定时，DoD 替你决定。

### 8.3 版本控制——Git + 迭代文件夹

你肯定用 Git 管理代码。美术资产也一样——**`.ase` 是源码，`.png` 是编译产物，两者都该纳入版本管理。** Git LFS 处理大的二进制 `.png` 文件更合适。

**迭代文件夹法：**

```
assets/sprites/hero/
├── v1/                      # 第一版
├── v2/                      # 第二版
├── v3/                      # 第三版
└── current/                 # 引擎从这里读取
```

`current/` 是引擎实际引用的路径。需要回退时，把旧版覆盖到 `current/`。迭代文件夹是你美术资产的"分支"——`current/` 是 `main`，`v1/v2/v3` 是 feature 分支，保留历史但不影响运行。

**铁律：永远不要用 "final" 这个词。** 它从来不是 final。用版本号——v1、v2、v3。版本号是递增的、客观的。`v3` 没骗你——它只是第三个版本。`final` 欺骗了你——它根本不是 final。

**Git 提交信息写成 "hero: redesign color palette v3"，** 而不是 "update"。就像你不会写 `git commit -m "fix"` 一样。

### 8.4 时间管理与瓶颈诊断

当你的资产生产速度不如预期时，不要笼统地说"我画得太慢了"。画得慢是一个症状，不是原因。

**四步诊断法：**

**第一步：记录一个资产的完整时间。** 用手机计时器记录每个阶段花了多少时间。

| 阶段 | 预计占比 | 实际耗时 | 偏差 |
|------|---------|---------|------|
| 参考收集 | 10% (6min) | 5min | OK |
| 草稿 | 15% (9min) | 3min | ⚠️偏少 |
| 粗胚 | 20% (12min) | 8min | ⚠️偏少 |
| 精修 | 35% (21min) | 50min | 🔴严重超标 |
| 导出 | 10% (6min) | 4min | OK |
| 引擎验证 | 10% (6min) | 2min | ⚠️偏少 |

**第二步：分析偏差。** 这个例子里精修花了 50 分钟而不是 21 分钟——因为草稿和粗胚被压缩了，你在定型不充分的情况下进入了细节，然后在精修阶段发现"比例不对""颜色要大改"。精修超支不是精修的问题，是前三阶段欠的债。

**第三步：对症下药。**
- **参考收集超支：** Pinterest 黑洞——刷了 40 分钟一张没用。解法：设 15 分钟硬时限。
- **精修超支（最常见）：** 你是在"磨"而不是在"画"——反复放大缩小看同一个地方、改一点又改回去。解法：每个精修阶段设 30 分钟上限，时间到就导出。如果不够好，记下来，下一个资产改进。
- **导出超支：** 每次导出都要手动调十几个参数。解法：保存导出预设（Aseprite + Godot 都支持）。
- **引擎验证被跳过：** "懒得导进引擎"。解法：引擎保持打开，资产导出后直接看效果。

**第四步：一次只修一个瓶颈。** 找到最拖你后腿的环节——通常就是精修超支——针对它做一个改变。下周再检查时间记录。

**返工公式：一个 bug 在阶段 N 发现，修复成本 = 在阶段 N-1 发现的 3-5 倍。** 这个数字不是精确计算，是一个经验方向——越晚发现，你要推翻的东西越多。这就是为什么草稿阶段不能跳过——在那里推翻一个方向只需要 5 分钟。

### 8.5 自动化——Lua 脚本 + CLI

规范让管线可复制，但手动执行规范本身需要时间。Aseprite 支持 Lua 脚本，能把重复性操作自动化。

#### 什么时候值得自动化？

一个简单的判断标准——按重复次数：

- **重复 3 次以上 →** 考虑用宏或快捷键简化
- **重复 10 次以上 →** 写 Lua 脚本
- **重复 100 次以上 →** 重新设计流程——这不是自动化能救的，是流程本身有问题

这和程序员写脚本的阈值一模一样——手动做两次可以忍，五次开始烦躁，十次必写脚本。

#### 脚本能帮你做什么

| 操作 | 脚本耗时 | 适用场景 |
|------|---------|---------|
| 批量导出所有图层的各自 PNG | 1 秒 | 角色分图层（身体/武器/特效） |
| 批量调色板替换 | 1 秒全文件 | 同一角色出白天/夜晚/四季版 |
| 批量整数倍缩放 | 1 秒全文件 | 所有资产统一缩放 |
| 自动 sprite sheet 打包 | 1 秒 | 多帧多方向打包为单张引擎用表 |
| 检测 Banding / 颜色超出色板 | 1 秒全文件 | 像素质检（Lint） |

**脚本存放位置：** `~/.config/aseprite/scripts/`（Linux/Mac）或 `%APPDATA%/Aseprite/scripts/`（Windows）。社区脚本库：https://github.com/aseprite/scripts。

#### CLI 命令行导出——不用打开 Aseprite

Aseprite 支持命令行批量操作。这对程序员来说是最自然的使用方式：

```
aseprite.exe -b hero.aseprite --sheet hero.png --data hero.json
```

`-b` = 后台模式（不打开 GUI），`--sheet` = 导出 Sprite Sheet，`--data` = 导出帧数据 JSON。把这个命令写进你的 build script——和 `npm run build` 一个道理。整套资产一键导出，不需要手动打开 Aseprite 一个一个点。

**Gold 原则：脚本只做重复性操作，创作性决策必须手控。** 批量导出是重复操作——脚本化。角色脸部的像素调整是创作决策——手控。

### 8.6 练习

#### L1 · 生命周期走一遍（20 分钟）

任选一个简单资产（如 16×16 药水瓶），按六阶段走一遍：参考→草稿→粗胚→精修→导出→引擎验证。最后把导出的 PNG 拖进 Godot，跑制作07 的 5 件检查。

**合格标准：** 一个能在引擎里显示的药水瓶 PNG，外加一张纸写清六个阶段各自的输入和输出。

#### L2 · 命名与版本控制审计（15 分钟）

打开你现有的项目文件夹。有没有文件名包含 `final`？有没有空格或中文？把违规的全部按 `类型_对象_动作_序号` 重命名，建 `v1/` + `current/` 迭代文件夹。

**合格标准：** 零个 `final`、零个空格、零个"看不出内容"的文件名。

#### L3 · 瓶颈计时（一周）

用 Toggl Track 或 Clockify，花一周记录每个资产的六阶段耗时。周末汇总成偏差表。找出最超支的阶段，做一个改变（如"精修设 30 分钟上限"）。第二周再记录，对比改善。

**合格标准：** 两周计时数据 + 偏差对比表 + 明确的下周改进目标。

### 8.7 本章小结

这一章讲的是一个看似无聊但决定你项目存亡的东西：**流程。**

```
需求 → 参考 → 草稿 → 粗胚 → 精修 → 导出 → 引擎验证 → Git → Release
  │       │    │     │      │     │    │         │       │
  │       │    │     │      │     │    │         │       └── 上线
  │       │    │     │      │     │    │         └── 版本记录
  │       │    │     │      │     │    └── QA：5件检查 + DoD
  │       │    │     │      │     └── 遵循制作07全部规范
  │       │    │     │      └── 最花时间——设30分钟上限
  │       │    │     └── 定型关键——比例定死后不改
  │       │    └── 2-3方案各5-10分钟——推翻成本最低
  │       └── 至少3-5张——具体到不需要再查
  └── 明确要画什么、尺寸多大、用途是什么
```

- **六阶段生命周期**不是官僚主义，是"不要在画到一半时才发现走错了方向"的保险单。
- **命名规范**不是强迫症——是三个月后你还能找到那个文件的唯一办法。
- **迭代文件夹 + Git**——`current/` 是 main 分支，`v1/v2/v3` 是历史记录。永远不用 `final`。
- **DoD（Definition of Done）：** 命名✓ 导出✓ 导入✓ 风格一致✓ Git提交✓ ——全部打勾 = Done。
- **瓶颈诊断 + 返工公式：** 一个 bug 在阶段 N 发现，修复成本是 N-1 的 3-5 倍。
- **自动化：** 重复 10 次以上写 Lua 脚本；CLI 命令行批量导出。脚本替代重复劳动，不替代审美决策。
- **导出阶段遵循制作07 全部规范。** 本章不再重复。

> **如果只记住一句话：** 你会画一个资产不等于你能生产一套资产——前者靠手，后者靠流程；六阶段生命周期 + 规范命名 + 迭代文件夹，是把"会画"翻译成"能生产"的三件套。
