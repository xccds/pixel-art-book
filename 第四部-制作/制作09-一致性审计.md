# 制作09 一致性审计：所有截图是一家人吗

### 9.0 这一章解决什么问题

制作08 你建好了生产管线——六阶段生命周期把"会画一个资产"翻译成"能生产一套资产"，命名规范、迭代文件夹、瓶颈诊断让你不会在 50 个资产里迷失自己。制作07 你解决了"单张资产进引擎"——Bilinear vs Nearest Neighbor、PNG 导出四件套、5 件导入检查、叠图对比。管线跑通了，每一张资产单独看都对。

但"每个资产独立看起来对"和"所有资产在一起看起来像同一个游戏"之间——有一道巨大的鸿沟。这道鸿沟的名字叫"视觉一致性"（Visual Consistency）。

视觉一致性是这样丧失的：2 月 3 日你画了第一个角色——色板暖色，线宽 2px，明度偏暗。2 月 17 日你画了第二个角色——你昨天刚看过一个高饱和的游戏——第二个角色的色板比第一个亮。3 月 5 日你画了一个 Boss——你刚学会一种新的渲染方法——Boss 的质感和你的第一个角色完全不同。4 月 12 日你画了 UI——这时候你不记得第一个角色的色板是什么颜色了——你的 UI 的颜色是"随机选的、好看就行"。

**结果：你的游戏有 50 个资产——如果把这 50 个 Sprite 从各自的场景中抽出来并排放——它们看起来像"5 个不同的游戏的资产被随意混在了一起"。** 但这不是你的错——这是人类大脑在"时间维度上保持视觉一致性"的生物学局限。你的审美感觉会随着你每天看的东西而变化——你不可能"记住"你 3 个月前用的线宽是多宽——就像你不可能"记住"你 3 个月前写的函数里的变量命名风格——除非你有 Lint 规则。

这一章是观察01 三能力模型里"**整**"（整合）能力的最后一层——全局验收层。观察01 说"整"是独立游戏开发者独有的顶层能力——纯美术从业者不需要考虑引擎导入和运行时一致性，但你必须考虑。制作07 解决"单张资产进引擎"，制作08 解决"一套资产怎么流水线生产"，这一章解决"这套资产放一起看起来是不是一家人"。三章合在一起，就是"整"的全部训练——也是第四部制作的收尾。

**本章核心承诺：** 你将拥有一张 10 项一致性审计检查清单——每一项是一个可执行的"是 / 否"判断——不需要"审美感觉"，只需要"看+回答"。你将看到 5 种最常见的视觉 Bug 的症状和修复——每一种配一张"症状截图 vs 修复后截图"。你将听到 Braid 开发者的经典一致性案例——如何在"非游戏性区域"减少细节以保护玩家的视觉注意力。你将学会"陌生人 3 秒测试"——这是你的审美能力被"习惯化"后的终极破幻觉工具。

### 9.1 10 项一致性审计检查清单——不是"审美评审"，是"Lint 检查"

在你的代码中——当你写了一段代码——你跑 Lint。Lint 不是"这个代码写得好不好"的美学判断——Lint 是"这个代码有没有明确的、可检测的错误"的自动检查。风格04 风格文档是你的风格的 TypeScript 类型定义。一致性审计清单——是你的风格的 Lint 规则。

**执行审计的三个前提条件：**
1. **你必须把截图并排。** 不是"在脑子里回忆"。把你游戏的所有截图的缩略图放在同一屏幕上——足够小（大概 200×150 像素的缩略图）——这个尺寸让你看到"整体一致性"而不是"单个细节"。
2. **执行时间是"做好 10-20 个资产后"。** 不是做完 50 个资产后——那时候要改已经太晚了。10-20 个资产是一个"能被审计的最小资产集"——你在这个阶段发现的 Bug——修复成本是"改 3 个文件"。如果 50 个资产后审计——修复成本是"改 15 个文件"。
3. **用"是 / 否"回答每个问题。** "大概 / 差不多 / 应该是" = "否"。一致性审计不容忍模糊——模糊 = 不一致的开始。

![一致性审计清单](images/制作/制作09-fig01-一致性审计清单.svg)

*图 制作09.1：一致性审计 10 项检查清单。每项的答案只能是"是"或"否"。如果某两个资产在某一项的答案不同——那项就是"不一致"的——修复它。注意后面 2 项用红色标出——说明在这个示例中第 7、8 项是需要修复的。*

**10 项清单逐项详解：**

**1. 线宽统一（Line Weight Consistent）——检查工具：放大到能看到单个像素**

在你的游戏中——所有 Sprite 的描边（Outline）/ 线条——宽度是否一致？如果角色 A 的边线是 2px——但角色 B 的边线是 1px——玩家可能不会"注意到"——但玩家的视觉系统会"感觉到"——这两个角色"不是用同样的工具画的"。线宽不统一 = "拼凑感"——等同于你的代码里有的函数用 camelCase，有的函数用 snake_case——编译器不会报错，但任何一个阅读你代码的人都会在第一秒感觉到"不干净"。线宽的像素级控制方法见练手01《线条》——阶梯原理与五种轮廓类型。

**2. 色板一致（Color Palette Consistent）——检查工具：Lospec 色板查看器 / 你的风格文档的色表**

你的所有 Sprite 使用的颜色——是否都来自你的风格文档定义的色板？如果你的色板定义了"红色 = #E74C3C"——而你在某个敌人身上用了"红色 = #FF4444"（因为你觉得"这个红更凶一点"）——你已经把你的色板"手动扩展"了一个新色——这个新色没有被记录在你的文档中。下一次你再用红色——你用的是文档中的 #E74C3C 还是你记忆中的 #FF4444？色板污染（Palette Pollution）是颜色一致性的头号杀手——它是渐进式的——第一天 1 个新色，第二天 1 个新色——第 90 天你的实际色板有 47 个颜色——但你的文档只记录了 16 个。调色板的选型与管理见练手05《色彩》——经典调色板（PICO-8 / DB16 / Endesga32）与色板替换流程。

**3. 细节密度连贯（Detail Density Consistent）——检查工具：缩略图并排法**

100×100 像素内的"线数 / 色块数 / 装饰元素数"——这些的总和叫"细节密度"。如果角色 A 有 5 个颜色、2 条线、1 个装饰——角色 B 有 12 个颜色、8 条线、4 个装饰——把这两个角色的缩略图并排——你不需要数，你的眼睛就能感觉到"左边简单右边复杂"——这就是细节密度跳跃（Detail Density Jump）。B 看起来像"另一个游戏的角色莫名出现在这个游戏中"。

**4. 抗锯齿方式统一（Anti-aliasing Uniform）——检查工具：放大看边缘像素**

你的 Sprite 的边缘——是有 AA（抗锯齿）的软边——还是没有 AA 的硬边？如果你的角色 A 的边缘是硬边（像素游戏的典型特征）——但角色 B 的边缘有 AA 的过渡像素（因为你在画 B 的时候开了画笔的 smooth 选项）——把这两个角色放在同一个背景上——A 的边缘"粘在地上"，B 的边缘"飘在半空"——因为 B 的 AA 过渡色在背景上制造了一个"微弱的半透明光晕"。抗锯齿的"何时用 / 方法 / 过度危害"完整展开见练手08《像素专属技法》——那一章也讲清楚了为什么"不一致的 AA 比没有 AA 更糟糕"。

**5. UI 风格融合（UI Integrates with World）——检查工具：遮住 HUD 看画面**

你的血条、按钮、文字——和你游戏画面的风格是"一个世界"的吗？如果你的画面是暖色低饱和的奇幻中世纪——但你的血条是亮蓝 + 亮红的霓虹色——你的 UI 在屏幕上"发光"——在说"我不是这个世界的一部分——我是贴在屏幕上的"。UI 的材质、形状、配色——必须和游戏世界共享相同的美术原则——否则 UI 在视觉上"高于"游戏世界——抢走了玩家对游戏世界的注意力。UI 的三功能 / 四类型 / 80-20 注意力分配 / 8 步工作流见制作04《UI 设计》——那里给出"UI 和世界共享同一套美术原则"的完整设计法。

**6. 帧率风格匹配（Frame Rate Style Matched）——检查工具：看两个动画相邻播放**

你的走路的动画是 4 帧/秒——你的攻击动画是 24 帧/秒——这两个动作"相邻播放"时——走路看起来像"幻灯片式的逐帧"，攻击看起来像"流畅的手绘动画"——帧率的不一致比画面风格的不一致更容易被感知——因为运动是视觉皮层的最古老输入（比色彩和形状更底层）——你的玩家在看到颜色之前已经"感觉到了"运动不一致。帧率选择、帧时长与 Disney 原则的像素适配见制作05《动画与特效》——帧矩阵（9 动作 × 帧数）是审计时的对照表。

**7. 光源统一（Light Source Direction Uniform）——检查工具：每个 Sprite 的高光 / 阴影方向**

如果你的角色 A 的光源在左上——头发的高光在左上，身体的阴影在右下——而角色 B 的光源在右上——这不是"两个角色站的位置不同"——这是"两个人用两种光源画的两个角色"——在玩家的视觉系统中——这产生了微认知冲突——"这个世界的光源到底在哪？"修复：所有 Sprite 使用同一定向光源，或至少统一"光源规则"（如"背面高光 + 正面暗面"）。光源规则应写进风格04 视觉风格文档的"光照"小节——审计时对照文档判定。

**8. 透视一致（Perspective Consistent）——检查工具：把所有 Sprite 放在同一个"视平面"上**

你的角色是正侧面视角（Pure Profile）——但你的宝箱是斜俯视（3/4 Top-Down）——这比你想象的更常见——因为"画正侧面的角色最容易"，"画俯视的宝箱最容易在 2D 中展示打开状态"。但结果是——你的角色面对的是"侧面镜头"——你的宝箱面对的是"俯视镜头"——两者在同一张 2D 截图中——你不需要学过透视——你的大脑从 2D 线索重建 3D 空间——大脑发现矛盾——报告为"看着有点怪但说不上在哪"。像素游戏的透视类型谱（正交 / 侧视 / 俯视 / 2:1 等距 / 塞尔达透视问题 / 45° dimetric / 斜视）见练手08《像素专属技法》——选型阶段就锁死一种透视，比审计时再统一要便宜得多。

**9. 比例协调（Proportions Coordinate）——检查工具：把两个 Sprite 的画布尺寸除以它们的"单位尺寸"**

你的角色是 3 头身（身高 = 3 个头的高度）——你的敌人是 5 头身——这是有意的吗？比例差异不是 Bug——如果是有意设计。但如果你的设计文档说"所有角色都是 2.5 头身的 chibi 比例"——而你做了一个 4.5 头身的 NPC——那这个 NPC 在"种族上"看起来属于另一个物理世界——不是属于你的游戏世界。

**10. 字体融入画面（Typography Integrates with World）——检查工具：在游戏截图中用"一个你没见过的文字"替换游戏文字**

如果你的游戏是像素风——但你的 UI 字体是 Arial（现代系统字体）——这个冲突和你把 Comic Sans 放在一份法律合同上是一个级别的风格冲突。字体是字体的"载色"——像素风对应 bitmap 字体（每个字符的像素由你手动定义），奇幻风对应衬线字体（serif），科幻风对应无衬线字体（sans-serif、等宽）。如果你没意识到"字体也是视觉风格的载体"——你的 UI 文字用默认系统字体——这是一个沉默的"不一致"——但它在你的玩家的"整体感觉"中是一个持续的重量。字体的像素适配属于制作04《UI 设计》的 UI 视觉语言范畴——审计时把它和 UI 风格融合一起判。

> **程序员类比：** 一致性审计 ≈ 你的代码的 CI（Continuous Integration）流水线。不是"审美评审"——是"自动化检查"。每条规则有明确的判定条件：线宽是否一致 = 检查 Sprite 的描边宽度是否全等于一个固定值。色板是否一致 = 检查 Sprite 的色值是否全在风格文档的数组中。当你的 CI 红了——你修。当你的审计清单红了——你也修。两者的心态是同一个东西：不相信"我自觉能做到"——相信"能自动检查出来的才是真的一致"。制作08 的迭代文件夹是版本控制层，这一章是 CI 层——两者合起来才是"整"的工程化。

### 9.2 5 种常见视觉 Bug——不是"不好看"，是"冲突"

以下是你在审计中最可能遇到的 5 种 Bug。每一种配症状描述、诊断方法和修复方法。

![5种视觉bug对比](images/制作/制作09-fig02-五种视觉bug对比.svg)

*图 制作09.2：5 种常见视觉 Bug 对比。左列 = 症状，右列 = 修复后。一个健康的视觉系统不是"所有资产都完美"——是"所有资产在同一个世界中"。这 5 种 Bug 的共同根源：某个资产在某个视觉维度上"出离"了风格文档的约束。*

**Bug 1：线宽不统一（Line Weight Inconsistency）**

- **症状：** 你把所有 Sprite 放一起——某些 Sprite 看起来"干净有力"（因为线粗），某些 Sprite 看起来"细弱模糊"（因为线细）。像用不同品牌的笔画同一张画——画出来的线条的"存在感"不同。
- **诊断方法：** 打开第一个 Sprite——测量一条边缘线的像素宽度。打开第五个 Sprite——测量同类型边缘的像素宽度。如果不同——线宽不统一。
- **修复：** 把线太粗的 Sprite——把粗线的外侧/内侧"削薄"到标准线宽。把线太细的 Sprite——在细线的外侧加上一层补齐标准线宽。修复不是"重画"——是"用标准线宽把线重新描一遍"。在 Aseprite 中——用 Contour 功能（Shift+O）可以把选区的轮廓以固定宽度描绘——这是修复线宽不统一最快的工具。线宽像素级控制的前置训练见练手01。

**Bug 2：色板污染（Palette Pollution）**

- **症状：** 某个 Sprite 中出现了你的色板中不存在的颜色——例如你的色板是暖色、低饱和——但某个敌人身上出现了一个高饱和的紫红色（因为你觉得"Boss 要显眼"）。这个颜色在你的其他 9 个 Sprite 中不存在。单个资产看不出来——但把所有 Sprite 并排就"突出来"——像一个音符音准不对。
- **诊断方法：** 在 Aseprite 中打开 Sprite——Color Palette 面板——对比你的风格文档的色板。任何不在文档中的颜色 = 色板污染。或者——用吸管工具取色——然后切换到另一个 Sprite——看这个色在另一个 Sprite 中是否存在。如果 5 个以上资产中都查不到 = 色板污染。
- **修复：** 把"外来色"替换为你的色板中最接近的色。如果"最接近的色"和你的意图差距太大——不是"这个外来色必须留下来"——是"你的色板少了一个必要色"。正式把你的色板扩展 1 个色——并把新色记录在你的风格04 风格文档中（加上"添加的原因"）——然后去所有的资产中确认是否需要把这个新色"回溯"应用到旧资产中。色板扩展的决策流程见练手05。

**Bug 3：细节密度跳跃（Detail Density Jump）**

- **症状：** 你把场景 A 的截图和场景 B 的截图并排——场景 A 的元素每个都只有 3-4 个颜色+简单的几何形——场景 B 的每个元素都有 8-10 个颜色+复杂的纹理+多处装饰。切换看两张图——像"两个分辨率的游戏"。
- **诊断方法：** 把两张缩略图（约 200×150）并排——一眼看过去——如果感觉"左边简单右边拥挤"——细节密度跳跃。不需要数字——眼睛的判断就是最准确的密度检测器。因为你的玩家的眼睛也是"一眼扫过去"——不是"花 5 秒分析"。
- **修复：** 确定一个"最大可接受细节密度"。选定一个复杂的 Sprite 作为"最复杂的资产"——其他所有资产不超过它的复杂度。或者——相反——选一个"最简资产"作为基线——把你最复杂的资产简化到接近基线的密度。是"提升简单资产"还是"简化复杂资产"——取决于你的目标风格——但规则是统一的：不能同时存在"极简"和"极繁"。

**Bug 4：抗锯齿不一致（Anti-aliasing Inconsistency）**

- **症状：** 角色 A 的边缘是"硬"的——每一个像素是色还是无色——没有灰色过渡。角色 B 的边缘有"软"的过渡像素——A 的颜色 → 浅灰 → 背景色。在画面中——A 和背景"粘"在一起——B 仿佛"飘浮"在背景前。
- **诊断方法：** 放大到 400%——看边缘。有灰像素 = AA 存在。如果没有灰像素 = 无 AA。如果两个 Sprite 的"有灰/无灰"不同——抗锯齿不一致。
- **修复：** 如果全部用硬边——在 Aseprite 中关掉 AA 选项（画笔工具的"Pixel-perfect"模式）——把所有有 AA 的 Sprite 的边缘灰色像素手动清除——换上纯色像素。如果全部用软边——确保你的 AA 参数在所有 Sprite 中一致（AA 的强度、AA 的颜色数量）——不一致的 AA 比没有 AA 更糟糕——因为它制造了"有的地方有抗锯齿、有的地方没有"的视觉矛盾。AA 的像素级操作与"过度 AA"反模式见练手08。

**Bug 5：UI 风格突变（UI Style Mutation）**

- **症状：** 战斗 UI 用的是金属质感（灰色渐变 + 铆钉），菜单 UI 用的是木纹质感（棕色平铺 + 花纹），设置 UI 用的是纯扁平（无质感）。把三张截图并排——像是三个不同的 UI 设计师做的——然后被硬嵌到了同一个游戏中。
- **诊断方法：** 截三张不同状态的游戏截图（战斗 / 菜单 / 对话）——只看 UI 部分（遮住游戏画面）。UI 的材质、配色、字体是否一致？如果三张截图的 UI 看起来像三个不同的应用——UI 风格突变。
- **修复：** 选定一个 UI 风格（参考风格04 风格文档中的 UI 规则）——把另外两个 UI 面板的"材质/配色/字体"统一到同一个风格。统一的 UI 不意味着"所有 UI 看起来一模一样"——意味着"所有 UI 看起来是由同一个人用同一套工具做出来的"。UI 风格选型与四风格谱见制作04。

### 9.3 案例研究：Braid——"一致性"不是都"一样多"

2008 年的《Braid》是独立游戏视觉设计史上的一座里程碑。开发者 David Hellman（美术 / 视觉设计）在他的开发日志中记录了一个关键的设计选择——这个选择完美阐释了你需要理解的"一致性"的真正含义。

**Braid 的视觉挑战：** Braid 是一个时间操控平台游戏——画面是"画家式"的手绘 2D（非像素艺术）。每个场景的背景极其丰富——多层视差滚动背景——充满复杂的笔触、纹理、渐变。在开发过程中，Hellman 面临一个问题：丰富的前景（角色、平台、可互动物体）和同样丰富的背景——在画面上产生了"视觉竞争"——玩家的眼睛花在"区分什么是可以踩的、什么只是好看的"上的时间——比花在"理解游戏"上的时间还长。

**Hellman 的解决方案：** "非游戏性区域（背景中玩家不可互动的部分）用更少的细节和更低的对比度。游戏性区域（平台、角色、可互动物体）可以用完整的细节密度。这样玩家的眼睛在 0.2 秒内就能区分'这是游戏'和'这是风景'。"

**这个方案对一致性的启示：** "一致性"不是"所有东西都一样精细"。一致性是——"你的不一致是有意设计的、是系统化的、是玩家能学会并依赖的"。Braid 的"游戏区域 vs 非游戏区域"的细节差异——不是"不一致的 Bug"——是"游戏设计的一部分"——相当于你代码中的"public method vs private method"——两者是"不同的可见性"——不是"风格不统一"。玩家在 Braid 中不需要读提示——他们从视觉上就知道"这个区域是游戏、那个区域是背景"——这是一种"视觉语言的一致性"——不是"所有像素看起来一样"的那种粗糙一致性。

> **对你作为独立开发者的应用——"有意的细节密度分层"：** 你的游戏的"可互动物体"可以用更粗的线宽、更鲜明的颜色、更多的帧——不是因为"要漂亮"，是因为"要可读"。你的背景元素可以用更细的线宽、更低的对比度、更少的帧——不是因为"不好看"，是因为"不能抢走玩家的注意力"。这两类元素的视觉差异不是 Bug——是"交互性编码"（Affordance Encoding）——你的玩家学会"粗线 = 可以踩 / 可以打 / 可以捡"——这个学习在 2-5 分钟游戏后就变成了肌肉记忆——从此不再干扰游戏。这条规则要写进风格04 风格文档——审计第 3 项"细节密度连贯"时按"分层规则"判，不是按"全部一样密"判。

### 9.4 陌生人 3 秒测试——你的"习惯化"的终极解药

你看了你的游戏 1000 遍。你的大脑已经把你的游戏画面"习惯化"了（Habituated）——你的视觉皮层在识别到"这是你的游戏"后——自动关闭了"视觉细节检测"——因为大脑假设"我已经看过这了，不需要再分析"。这就像你住在一个房间 3 年了——不再注意到墙上那一道从第 1 天就有的裂缝——直到一个客人进门说"你那道裂缝有点大"——你才"啊？有裂缝？"——因为你已经自适应了。

**你的"习惯化"让以下问题对你"隐形"了：**
- 你的游戏在两个场景之间的颜色有点不同——但你 30 秒就习惯了——你的玩家不会。
- 你的 UI 字体在某个面板突然变小了——但你 30 秒就习惯了——你的玩家会觉得"这个面板不对"。
- 你的角色在走路时的帧有点不均匀——但你看了 5000 遍——你的大脑已经"修复"了这个不均匀——把原本不均匀的感觉"平均"成了均匀——这是大脑的感知补全（Perceptual Completion）——非常强大——非常危险。

**陌生人 3 秒测试的操作：**
1. 找一个能来你电脑前的人——不是游戏开发者，不是你的玩家群体——就是一个普通人。家人、室友、外卖员——都行。
2. 打开你游戏的一张有代表性的截图——战斗画面、主场景画面、角色在场景中站立的画面——全屏显示。
3. 说："你不需要理解这个游戏——你只需要看这张图 3 秒。"（在 3 秒后关掉图）。
4. 问："请说 3 个词描述你看到的画面。"不要问"你感觉怎么样"——问"描述"。他们要输出的不是评价——是观察。
5. 你记下这三个词。然后对照风格04 你写在风格文档中的 3-5 个关键词。
6. 如果这 3 个词和你文档的关键词**高度重叠**（比如你的关键词是"温暖 · 简洁 · 童趣"——对方的词是"暖和、卡通的、可爱"）——你的视觉一致性在传递正确的信息。
7. 如果对方说的 3 个词和你的关键词**没有重叠**（比如你的关键词是"温暖 · 简洁 · 童趣"——对方的词是"乱、灰的、看不清楚"）——你的画面在传达与你的意图完全相反的信息。这不意味着对方的审美够不上——这意味着你的视觉语言的"传达准确性"严重不足。回去修正——再用不同的人再测——直到至少有一半的陌生人说的 3 个词中有至少 1 个和你的关键词匹配。

**这个测试的科学原理：** 3 秒是人脑在不分析的状态下"第一印象"的时间窗口。第一印象的 3 个词——是这个画面存储在被试者的"本能层"记忆中的东西——不是"思考后"的东西。你真正在测的是——"你的游戏画面的'本能传达力'——和你的设计意图是否一致"——这是一个比"我觉得好看"有量级更高的诊断价值。

### 9.5 一致性——你最后的"质量门"

在代码中——合并前你做 Code Review。一致性审计 = 你的视觉 Code Review。这不是"审美品味测试"——这是"条目标志"——"这个资产——在这个视觉维度上——离群了吗？"

你在以下时机执行一致性审计：
- **第 10 个资产之后**（这个量足够体现"一致性有没有问题"）。
- **加入一个新类型的资产时**（你之前只有角色——现在你加了第一个敌人——敌人看起来和玩家角色是"一个世界"的吗？）。
- **加入 UI 后**（UI 是"最容易离开游戏世界"的视觉元素——因为它很多时候是在独立的图层上做的）。

如果你能在第 20 个资产时让一致性审计的 10 项都"通过"——你的游戏在视觉上是"一家人"的——不是"一群人被各自画好然后塞进同一个屏幕"。这个成就——比你"画得好看"——重要得多——因为"一致性"是你的——"好看"是你不能完全控制的。**"一致性"是"工程"——"好看"是"彩票"——但一个"一致但不太好看"的游戏在玩家眼中的效果——在评分、体验、沉浸感上——优于一个"零散时好看但整体混乱"的游戏。**

### 9.6 练习——做你自己的审计官

#### L1 · 第一次 10 项审计（30 分钟）

1. 把你游戏中的 10-20 个资产截图——做成缩略图（约 200×150 每个）——用任何看图软件排列在一个屏幕上。
2. 拿一张纸（或一个新笔记文件）——写下 10 项的检查标题。
3. 逐项执行是/否判断。对每一项——把你的"最不一致的两个资产"挑出来——放在旁边看——确认"是——它们在这一项不一致"还是"否——我眼花了，它们是一致的"。
4. 终点不是"10/10"——是"每一项的'否'都有一个明确的修复计划"。比如"第 7 项：光源不统一。修复计划：把角色 2 和角色 5 的光源从右侧翻转到左侧。"把这个修复计划记在你的 Bug Tracker 或 Task List 中——给你自己安排一个修复时间。

**合格标准：** 一张填完的 10 项审计表——每一项有明确的"是/否"判定——每一个"否"配一条具体的修复计划（含目标资产 + 修复动作 + 预计耗时）。不是"我看了觉得还行"，是"我能拿出一张纸指给你看哪几项红、哪几项绿"。

#### L2 · 第一次 3 秒陌生人测试（10 分钟）

1. 找一个人。
2. 开你游戏的最有代表性的一张截图——3 秒。
3. 记录对方说的 3 个词。
4. 打开你的风格04 风格文档——看你的 3-5 个关键词。对比。如果重叠 < 1 个——思考为什么——在截图中有没有某些视觉元素在"争夺"对方的注意力——例如你做了大量的装饰——但对方根本没看到你的角色的动作——因为背景的花纹太显眼。回去修正——再测——直到重叠 ≥ 1。

**合格标准：** 至少 3 个陌生人的 3 词反馈记录 + 每次反馈与风格文档关键词的重叠数。如果 3 次中有 2 次重叠 ≥ 1——合格。如果 3 次中 0 次重叠——不要继续测人，回到 L1 重新审计——问题不在测试方法，问题在画面本身。

#### L3 · 开一个"一致性 Bug 列表"

在你的 Bug Tracker 或笔记中——专门开一个列表——名字叫"视觉一致性 Bug"。每一个条目记录：资产名 + 不一致的维度 + 症状描述 + 修复状态。当你做一致性审计时——把新发现的条目加到这个列表中。当你的修复做完——"关 Bug"。视觉一致性的"Bug 列表" = 你的 CI 的红灯板——它给你可见的进度——从"很多红灯"到"一片绿色"。

**合格标准：** 一个持续维护的"视觉一致性 Bug"列表——至少 5 个条目经过"发现→修复→关闭"的完整生命周期。这个列表在你后续每一个新项目里继续用——它是你"整"能力的可累积资产。

### 9.7 本章小结

这一章是第四部制作的收尾，也是观察01 三能力模型里"整"（整合）能力的最后一层——全局验收。

- **10 项一致性审计清单**（线宽 / 调色板 / 细节密度 / 抗锯齿 / UI 融合 / 帧率 / 光源 / 透视 / 比例 / 字体）不是"审美评审"——是"Lint 检查"。每项只能答"是/否"，模糊等于"否"。像素专属项的深度训练分布在练手01（线宽）、练手05（调色板）、练手08（抗锯齿 + 透视）、制作04（UI + 字体）、制作05（帧率）——审计时遇到红灯，回对应章补课。
- **5 种常见视觉 Bug**（线宽不统一 / 色板污染 / 细节密度跳跃 / 抗锯齿不一致 / UI 风格突变）每一种配症状 + 诊断 + 修复。共同根源：某个资产在某个视觉维度上"出离"了风格04 风格文档的约束。
- **Braid 案例给出"一致性"的真正含义**——不是"所有东西都一样精细"，是"不一致是有意设计的、系统化的、玩家能学会并依赖的"。交互性编码（粗线 = 可互动 / 细线 = 背景）是合法的"分层一致性"。
- **陌生人 3 秒测试**是"习惯化"的终极解药——你看了 1000 遍的画面，大脑已经"感知补全"掉了所有瑕疵；陌生人的第一印象 3 个词，是你看不到的真相。3 秒测试的科学基础是感知补全心理学。
- **质量门**在第 10 个资产后、加入新类型资产时、加入 UI 后三个时机执行。能在第 20 个资产时 10 项全绿——你的游戏在视觉上是"一家人"。

> **如果只记住一句话：** "一致性"是"工程"——"好看"是"彩票"——一个"一致但不太好看"的游戏在玩家眼中优于一个"零散时好看但整体混乱"的游戏；你的工作不是画出好看，是让所有资产在同一个视觉世界里。

> **上手行动：** 今晚做 L1——30 分钟把你游戏的 10-20 个资产做成缩略图并排，跑一遍 10 项审计，把每一个"否"配一条修复计划。明天做 L2——10 分钟找一个陌生人做 3 秒测试，对比你的风格04 关键词。开一个 L3 的"视觉一致性 Bug"列表——从今天起每一个新资产都过审计。
